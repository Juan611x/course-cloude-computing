# Amazon Machine Images (AMIs)

Cuando lanzas una instancia EC2, AWS necesita saber con qué sistema operativo arrancarla, qué
software debe tener preinstalado, cómo debe estar configurado el disco y qué permisos de acceso
aplicar desde el primer segundo. Todo eso viene encapsulado en una **Amazon Machine Image (AMI)**
— una plantilla digital que define el estado inicial completo de una máquina virtual.

Piénsalo así: si una instancia EC2 es la computadora encendida y corriendo, la AMI es la imagen
de fábrica que define exactamente cómo viene configurada desde la caja. Cada AMI contiene:

- El **sistema operativo** (Linux, Windows, macOS)
- El **software preinstalado** (servidores web, bases de datos, agentes de monitoreo)
- La **configuración del volumen raíz** (tipo de disco, tamaño inicial, sistema de archivos)
- Los **permisos de lanzamiento** (quién puede usar esa AMI)

## Caso de uso: el problema de configurar desde cero

Imagina que tienes una aplicación Node.js en producción y necesitas escalar horizontalmente —
pasar de una instancia a cinco durante un pico de tráfico. Sin AMIs, tendrías que:

1. Lanzar cinco instancias vacías
2. Conectarte a cada una por SSH
3. Instalar Node.js, tu gestor de paquetes, las dependencias
4. Copiar el código, configurar variables de entorno, ajustar permisos
5. Iniciar el servicio en cada una

Cinco veces. Con riesgo de error humano en cada paso. Con tiempos de configuración de 10-20
minutos por servidor.

Con una AMI personalizada el flujo es completamente diferente: capturas el estado de tu servidor
ya configurado, creas una AMI a partir de él, y cuando necesitas escalar lanzas cinco instancias
desde esa AMI. En segundos, sin intervención manual, todas idénticas.

::: info AMIs vs imágenes de Docker
Si conoces Docker, el concepto de "imagen" puede generar confusión. Son similares conceptualmente
pero operan a niveles completamente distintos:

- Una **imagen Docker** empaqueta una aplicación y sus dependencias a nivel de proceso. Corre
  sobre un kernel compartido del sistema operativo del host.
- Una **AMI** empaqueta un sistema operativo completo con todo lo que corre sobre él. Es la base
  sobre la que podrías, de hecho, instalar Docker y sus imágenes.

En términos prácticos: una AMI es la máquina entera; una imagen Docker es una aplicación dentro
de esa máquina.
:::

## Tipos de AMI

### AMIs de AWS (Quick Start)

Son las AMIs oficiales creadas y mantenidas directamente por AWS. Representan la opción más
recomendada para empezar porque:

- **Siempre están actualizadas** — AWS publica nuevas versiones con parches de seguridad
  periódicamente
- **Son auditadas y firmadas** — puedes verificar su autenticidad con el _Verified provider_ badge
- **Tienen soporte garantizado** — Amazon Linux 2023, por ejemplo, incluye 5 años de soporte
  de largo plazo
- **Son gratuitas** — no tienen costo adicional más allá del uso de la instancia

![Panel de Quick Start AMIs en EC2](/assets/02-compute/03-amis/ami-quick-start.png)

Al crear una instancia, el panel **Quick Start** muestra los sistemas operativos más usados:
Amazon Linux, Ubuntu, Windows Server, Red Hat, SUSE Linux, Debian y macOS. Al seleccionar
cualquiera aparece la AMI más reciente recomendada, con sus detalles: ID, arquitectura soportada,
modo de boot, fecha de publicación y el usuario SSH predeterminado para conectarte.

### AMIs del Marketplace

El **AWS Marketplace** es una tienda de software donde empresas de terceros publican AMIs con
sus productos ya configurados. Son ideales cuando necesitas una solución específica lista para
usar sin pasar horas instalando y configurando software manualmente.

![AWS Marketplace AMIs](/assets/02-compute/03-amis/ami-marketplace.png)

Puedes acceder al Marketplace desde dos lugares:

- Directamente desde el panel de creación de instancias → botón **Browse more AMIs**
- Desde el panel principal de EC2 → sección **Images → AMI Catalog**

Las AMIs del Marketplace tienen **modelos de precio mixtos**: algunas son gratuitas (solo pagas
por la instancia EC2 que usas), otras tienen un costo de software adicional que se factura
junto con la infraestructura de AWS.

![Detalle de precios de una AMI de Marketplace](/assets/02-compute/03-amis/ami-marketplace-pricing.png)

El Marketplace suele ofrecer una **prueba gratuita** (típicamente 30 días) para que puedas
evaluar el software antes de comprometerte. Una vez pasado ese período, el cargo se aplica
por instancia activa — ya sea por hora o como contrato a plazo (30, 90, 365 días). Los
contratos a plazo generalmente ofrecen descuentos de hasta el 69% respecto al precio por hora.

::: warning Costos ocultos en el Marketplace
Antes de lanzar una instancia desde una AMI de pago, verifica el precio del software
**por separado** del precio de EC2. Una AMI empresarial puede costarte más que la instancia
en sí. El panel de precios del Marketplace muestra el costo desglosado por tipo de instancia.
:::

### AMIs de la Comunidad

Son AMIs creadas y mantenidas por usuarios individuales o la comunidad open source, no por
empresas. El concepto es el mismo que el Marketplace — AMIs preconfiguradas con software
específico — pero sin el proceso de verificación y soporte comercial que tienen las AMIs
de Marketplace.

Son útiles para configuraciones muy específicas o software de nicho que nadie empaqueta
comercialmente, pero requieren más cuidado: **verifica siempre la fuente antes de usarlas
en producción**. Una AMI de comunidad podría tener software desactualizado, configuraciones
inseguras o, en el peor caso, código malicioso.

### AMIs Propias (Custom AMIs)

Son las AMIs que tú mismo creas a partir de una instancia existente. Son el núcleo de la
automatización en AWS: capturas el estado exacto de un servidor ya configurado y lo conviertes
en una plantilla reutilizable.

> La visibilidad de las AMIs propias es **por región** — una AMI creada en `eu-west-1` no
> aparece en `us-east-1`. Si necesitas usarla en otra región, debes copiarla explícitamente
> desde **Actions → Copy AMI**.

Las veremos en detalle en la última sección de este capítulo.

## Consultar AMIs desde la CLI

Para listar las AMIs de Amazon Linux 2023 disponibles y obtener la más reciente:

```bash
aws ec2 describe-images \
  --owners amazon \
  --filters "Name=name,Values=al2023-ami-*" \
  --query "Images | sort_by(@, &CreationDate) | [-1].{ImageId: ImageId, Name: Name}" \
  --output table
```

**Desglose del comando:**

| Parte                                       | Qué hace                                                                                   |
| ------------------------------------------- | ------------------------------------------------------------------------------------------ |
| `--owners amazon`                           | Filtra solo AMIs publicadas por AWS (evita resultados de terceros)                         |
| `--filters "Name=name,Values=al2023-ami-*"` | Aplica un filtro por nombre con wildcard — solo AMIs cuyo nombre empiece con `al2023-ami-` |
| `Images \| sort_by(@, &CreationDate)`       | Ordena la lista completa por fecha de creación (JMESPath)                                  |
| `\| [-1]`                                   | Toma el último elemento, que es el más reciente                                            |
| `.{ImageId: ImageId, Name: Name}`           | Proyecta solo los campos que nos interesan                                                 |
| `--output table`                            | Formatea la salida como tabla legible                                                      |

::: tip Adaptar el filtro a otros sistemas operativos
Cambia el patrón en `Values=` para buscar otras AMIs:

- Ubuntu: `Values=ubuntu/images/hvm-ssd/ubuntu-*`
- Windows Server 2022: `Values=Windows_Server-2022-English-Full-Base-*`
- Amazon Linux 2 (legacy): `Values=amzn2-ami-hvm-*`
  :::

## Fases esenciales del ciclo de vida

Para entender cómo funcionan las AMIs — y más adelante cómo crearlas — necesitamos revisar
tres fases específicas del ciclo de vida de una instancia. Ya vimos el ciclo completo en
el capítulo de EC2; aquí nos enfocamos solo en las fases donde la AMI y su configuración
tienen protagonismo.

### Fase 1 — Creación (_Provisioning_)

Es el momento en que AWS construye la infraestructura base de la instancia: reserva capacidad
de cómputo en el hardware físico, asigna la interfaz de red, el volumen de almacenamiento
raíz y los identificadores únicos (instance ID, IP privada). En esta fase, la **AMI define
qué se debe instalar** — es la plantilla que AWS lee para saber con qué imagen de disco
arrancar la máquina.

_Aprovisionar_ en este contexto significa asignar y preparar los recursos que la instancia
necesita para existir: CPU, RAM, red y disco. La instancia pasa del estado `pending` al
estado `running` cuando el aprovisionamiento termina.

### Fase 2 — Configuración (_Bootstrap_)

El servidor ya existe y ha arrancado. Ahora se ejecutan los pasos de configuración inicial:
instalar paquetes adicionales, copiar archivos de configuración, ajustar servicios, crear
usuarios. Aquí entra en juego el **User Data**.

**¿Qué es User Data?**
Es un script (bash, PowerShell o cloud-init) que le pasas a la instancia en el momento de
lanzarla y que se ejecuta automáticamente durante el arranque. Puedes usarlo para:

- Instalar software (`yum install -y nginx`)
- Descargar código desde un repositorio
- Registrar la instancia en un sistema de monitoreo
- Configurar variables de entorno

::: info ¿User Data se ejecuta solo la primera vez?
**Por defecto, sí — solo en el primer arranque** después de lanzar la instancia. Si reinicias
la instancia, User Data no se vuelve a ejecutar automáticamente.

Sin embargo, puedes configurarlo para que se ejecute en cada arranque modificando el archivo
`/etc/cloud/cloud.cfg` (en Amazon Linux) y cambiando `scripts-user` por `always`. Esto es
útil, por ejemplo, para tareas de sincronización de configuración que deben ocurrir cada vez
que la instancia levanta.
:::

### Fase 3 — Ejecución (_Running_)

El servidor está funcionando con todas sus aplicaciones y servicios listos. En esta fase AWS
expone la **metadata de instancia** — un endpoint HTTP interno accesible únicamente desde
dentro de la instancia que proporciona información en tiempo real sobre ella:

```bash
# Desde dentro de la instancia
curl http://169.254.169.254/latest/meta-data/

# Ejemplos útiles:
curl http://169.254.169.254/latest/meta-data/instance-id
curl http://169.254.169.254/latest/meta-data/public-ipv4
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
```

La IP `169.254.169.254` es una dirección **link-local** — solo existe dentro de la red de
AWS y nunca sale a Internet. Está disponible en todas las regiones y tipos de instancia.

## Crear AMIs personalizadas

AWS ofrece dos rutas principales para crear tus propias AMIs: un servicio dedicado llamado
**EC2 Image Builder** diseñado para pipelines automatizados, y el método más directo de
**crear una AMI desde una instancia existente** ya configurada a mano. Ambas parten desde
el panel de AMIs en EC2.

![Panel de AMIs en EC2 con acceso a Image Builder](/assets/02-compute/03-amis/ami-console-panel.png)

Desde el panel lateral de EC2, en la sección **Images → AMIs**, puedes ver tus AMIs propias
y acceder al botón **EC2 Image Builder** en la barra superior para ir al servicio de pipelines.

### EC2 Image Builder

EC2 Image Builder es un **servicio completamente independiente de EC2** — no es una función
dentro de la consola de EC2, sino su propio servicio con su propio panel. Puedes acceder a él
de dos formas: desde el botón en el panel de AMIs o directamente buscándolo por nombre en la
barra de búsqueda global de AWS.

![Página de inicio de EC2 Image Builder](/assets/02-compute/03-amis/ami-image-builder-landing.png)

![Búsqueda directa de Image Builder en la consola](/assets/02-compute/03-amis/ami-image-builder-search.png)

Como se ve en la búsqueda, **EC2 Image Builder aparece bajo su propio servicio**, separado de
EC2. Esto es importante porque sus recursos (pipelines, recetas, componentes) tienen su propio
ciclo de vida y facturación independiente — solo pagas por las instancias que el pipeline usa
durante la construcción, no por el servicio en sí.

Image Builder es útil cuando necesitas un proceso repetible y auditado: define exactamente qué
instala, ejecuta pruebas de validación automáticas, y puede distribuir la AMI resultante a
múltiples regiones en un solo paso. Si solo necesitas crear una AMI ocasionalmente desde un
servidor que ya configuraste a mano, el método de instancia (explicado más adelante) es más
práctico.

#### Flujo de creación de un pipeline

La creación de un pipeline en Image Builder sigue seis pasos. Los repasamos de forma general
— los detalles avanzados (integraciones con Systems Manager, distribución multi-cuenta) se
cubren en módulos posteriores.

**Paso 1 — Detalles del pipeline**

![Paso 1: Detalles del pipeline](/assets/02-compute/03-amis/ami-builder-step1-pipeline.png)

Define el nombre del pipeline, su descripción y con qué frecuencia se ejecuta (manual,
programado con cron, o activado por cambios en componentes). También configuras el bucket
de S3 donde Image Builder guardará los logs de construcción.

![Paso 1: Configuración de logging](/assets/02-compute/03-amis/ami-builder-step1-logging.png)

**Paso 2 — Receta (_Recipe_)**

![Paso 2: Selección de receta](/assets/02-compute/03-amis/ami-builder-step2-recipe.png)

Una **receta** es la definición de qué contiene tu imagen: qué AMI base usar como punto de
partida, qué componentes instalar, y cómo configurar el almacenamiento. Puedes reutilizar una
receta existente o crear una nueva. El panel de _Recipe Details_ muestra el resumen de la
receta seleccionada — si está vacío es porque aún no has creado ninguna.

Al crear una nueva receta, lo primero es elegir el tipo de imagen de salida:

![Paso 2: Nueva receta — tipo de imagen](/assets/02-compute/03-amis/ami-builder-step2-new-recipe.png)

Puedes crear una **AMI** (para lanzar instancias EC2) o un **contenedor Docker** (para publicar
en Amazon ECR). La opción de Docker es útil cuando quieres usar Image Builder como pipeline
de CI/CD para imágenes de contenedor, aprovechando sus capacidades de pruebas y distribución
automática. En este capítulo nos enfocamos en AMIs.

Luego defines los **componentes** de la receta, divididos en dos tipos:

![Paso 2: Componentes de build y test](/assets/02-compute/03-amis/ami-builder-step2-components.png)

Los **Build Components** son los pasos que se ejecutan para construir la imagen: instalar
paquetes, ejecutar scripts, copiar archivos de configuración. Los **Test Components** son
pruebas que se ejecutan sobre la imagen ya construida para validar que todo funciona antes de
aprobarla — por ejemplo, verificar que un servicio arranca correctamente.

También configuras el almacenamiento del volumen raíz:

![Paso 2: Configuración de almacenamiento](/assets/02-compute/03-amis/ami-builder-step2-storage.png)

**Paso 3 — Proceso de creación de la imagen**

![Paso 3: Workflows de creación](/assets/02-compute/03-amis/ami-builder-step3-workflow.png)

Define el flujo (_workflow_) que Image Builder seguirá para construir la imagen: lanzar
una instancia temporal, ejecutar los componentes de build, crear un snapshot, ejecutar los
componentes de test sobre una segunda instancia, y finalmente aprobar o rechazar la imagen.

**Paso 4 — Configuración de infraestructura** _(opcional)_

![Paso 4: Infraestructura](/assets/02-compute/03-amis/ami-builder-step4-infra.png)

Puedes especificar qué tipo de instancia EC2 debe usar el pipeline para construir la imagen,
en qué VPC y subnet, y qué rol IAM debe asumir. Si lo dejas en los valores por defecto,
Image Builder usa una instancia `t3.medium` en la VPC default de la región.

**Paso 5 — Configuración de distribución** _(opcional)_

![Paso 5: Distribución](/assets/02-compute/03-amis/ami-builder-step5-distribution.png)

Configura a qué regiones y cuentas AWS se debe distribuir la AMI resultante después de que
el pipeline la apruebe. Esto automatiza el proceso de copiar la AMI entre regiones, que de
otro modo tendrías que hacer manualmente.

**Paso 6 — Revisión**

![Paso 6: Revisión del pipeline](/assets/02-compute/03-amis/ami-builder-step6-review.png)

::: danger Alerta de seguridad — Security Scanning deshabilitado
En la página de revisión, Image Builder muestra una advertencia visible: **Inspector scans
are not enabled for this pipeline**. Esto significa que la imagen que construyas **no será
analizada en busca de vulnerabilidades CVE** antes de ser aprobada.

Para entornos de producción, activa Amazon Inspector en tu cuenta y configura el pipeline
para ejecutar el escaneo automático. Una AMI con vulnerabilidades conocidas expone todas las
instancias que lances a partir de ella.
:::

Una vez revisado y confirmado, el pipeline queda listo para ejecutarse. Puedes lanzarlo
manualmente desde el panel de Image Builder o esperar al intervalo programado.

::: info Documentación oficial
Este capítulo cubre el flujo general de Image Builder. El servicio tiene muchas más
capacidades: gestión de versiones de recetas, componentes personalizados con AWSTOE, pruebas
de cumplimiento con Security Hub, y distribución multi-cuenta con AWS Organizations. Consulta
la [documentación oficial de EC2 Image Builder](https://docs.aws.amazon.com/imagebuilder/latest/userguide/what-is-image-builder.html)
para el detalle completo.
:::

### Creación basada en instancia desde la consola

El método más práctico para la mayoría de casos es más directo: levantas una instancia EC2,
la configuras exactamente como necesitas — instalas software, ajustas configuración, preparas
el entorno — y luego creas una AMI a partir de esa instancia. Es el flujo favorito porque
permite prueba y error real antes de "congelar" el estado en una imagen.

El comando CLI para crear la AMI es:

```bash
aws ec2 create-image \
  --instance-id i-0ca8100a66d2d389 \
  --name "mi-servidor-web-v1" \
  --description "Servidor web con Nginx + Node.js 20 configurado" \
  --no-reboot
```

El flag `--no-reboot` le indica a AWS que **no reinicie la instancia** durante el proceso de
captura. Sin él, AWS reiniciaría la instancia para garantizar la consistencia del sistema de
archivos — útil si quieres una imagen con sistema de archivos limpio, pero interrumpe el
servicio. Con `--no-reboot` la instancia sigue corriendo, aunque hay un pequeño riesgo de
inconsistencia en archivos que estaban siendo escritos en ese momento.

El comando devuelve el ID de la nueva AMI:

![Salida del comando aws ec2 create-image](/assets/02-compute/03-amis/ami-cli-create-output.png)

Para verificar que la AMI quedó disponible puedes consultarla desde la CLI:

```bash
aws ec2 describe-images \
  --owners self \
  --query "Images[*].{ID: ImageId, Name: Name, State: State, Created: CreationDate}" \
  --output table
```

O directamente desde la consola en **EC2 → Images → AMIs**, filtrando por _Owned by me_:

![Listado de AMIs propias en la consola](/assets/02-compute/03-amis/ami-console-list.png)

#### Copiar una AMI a otra región

Recuerda que las AMIs son regionales — una AMI en `eu-west-1` no aparece en `us-east-1`. Para
copiarla explícitamente entre regiones, usa el comando `copy-image`:

```bash
aws ec2 copy-image \
  --source-region eu-west-1 \
  --source-image-id ami-0abcdef1234567890 \
  --region us-east-1 \
  --name "mi-app-servidor-v1.0-us"
```

Aquí `--source-region` y `--source-image-id` identifican la AMI de origen, y `--region`
indica el destino. El comando crea una nueva AMI en la región destino con el nombre especificado
— los snapshots también se copian automáticamente.

#### Lanzar una instancia desde tu AMI

Con la AMI lista, puedes lanzar instancias desde ella directamente desde el panel de AMIs:
seleccionas la AMI y usas el botón **Launch instance from AMI**:

![Detalle de AMI con opción de lanzar instancia](/assets/02-compute/03-amis/ami-console-detail.png)

Esto abre el flujo estándar de creación de instancias EC2, pero con tu AMI preseleccionada
como imagen base. Cualquier configuración adicional (tipo de instancia, security groups,
key pair) la defines en ese momento.

#### Lanzar desde CLI

El comando es el mismo `run-instances` que ya conoces — la única diferencia es que en
`--image-id` pones el ID de tu AMI en lugar de una AMI pública:

```bash
aws ec2 run-instances \
  --image-id ami-01a189fb2a6db4faf \
  --instance-type t3.micro \
  --key-name mi-keypair \
  --security-group-ids sg-0abc123 \
  --count 1
```

Si quieres lanzar directamente desde la AMI más reciente que creaste, puedes encadenar
`describe-images` para obtener el ID sin tener que copiarlo manualmente:

```bash
AMI_ID=$(aws ec2 describe-images \
  --owners self \
  --query "Images | sort_by(@, &CreationDate) | [-1].ImageId" \
  --output text)

aws ec2 run-instances \
  --image-id "$AMI_ID" \
  --instance-type t3.micro \
  --key-name mi-keypair \
  --security-group-ids sg-0abc123 \
  --count 1
```

Esto es útil en pipelines de despliegue: Image Builder termina, crea la AMI, y el siguiente
paso del pipeline la lanza automáticamente sin hardcodear ningún ID.

### Ciclo de vida de tus AMIs

Una vez que ya no necesitas una AMI, puedes darla de baja en dos pasos desde el menú
**Actions** del panel de AMIs:

![Menú de acciones de una AMI](/assets/02-compute/03-amis/ami-console-actions.png)

1. **Deregister AMI** — la elimina del catálogo para que no pueda usarse para lanzar nuevas
   instancias. Las instancias ya corriendo no se ven afectadas.
2. **Delete associated snapshots** — libera el almacenamiento (los snapshots se cobran por GB/mes
   mientras existan, independientemente de si la AMI sigue registrada o no).

::: warning Desregistrar no elimina los snapshots
Si solo desregistras la AMI sin borrar sus snapshots, seguirás pagando por el almacenamiento
de esos snapshots indefinidamente. Siempre verifica en **EC2 → Elastic Block Store → Snapshots**
que no quedaron snapshots huérfanos después de dar de baja una AMI.
:::
