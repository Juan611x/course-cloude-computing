# UserData y Metadata en EC2

Cuando lanzas una instancia EC2, el sistema operativo arranca vacío — sin tu software, sin tu
configuración, sin nada de lo que tu aplicación necesita. Para llenar ese vacío tienes dos
mecanismos complementarios que funcionan en momentos distintos del ciclo de vida:

**UserData** opera en el arranque: le dices a la instancia _qué debe hacer_ la primera vez que
enciende. Es una vía de entrada — tú le mandas instrucciones a ella.

**Metadata** opera durante la ejecución: la instancia _consulta información sobre sí misma_ en
tiempo real — su IP, su región, el rol IAM que tiene asignado. Es una vía de salida — ella te
responde a ti.

Son dos caras del mismo concepto: datos que viven en la frontera entre la instancia y la
infraestructura de AWS.

---

## UserData

Imagina que estás montando un Auto Scaling Group que debe escalar de 2 a 10 instancias durante
un pico de tráfico. Cada instancia nueva tiene que tener Nginx instalado, el código de tu
aplicación descargado desde S3, las variables de entorno configuradas y el servicio arrancado.
Sin UserData tendrías que conectarte por SSH a cada una y hacerlo manualmente — inviable a
escala. Con UserData, ese proceso ocurre automáticamente en el momento del arranque, sin
intervención humana, en cada instancia que el grupo lance.

**UserData es un script que le pasas a la instancia en el momento de crearla y que se ejecuta
automáticamente durante el primer arranque.** Es la herramienta principal para automatizar la
configuración inicial de tus servidores. Sus casos de uso más frecuentes son instalar paquetes
y dependencias del sistema, configurar y arrancar servicios, descargar artefactos desde S3 o
clonar repositorios de Git, y registrar la instancia en sistemas externos como un Load Balancer
o un servidor de configuración centralizado. Este último caso —el registro automático en un
Load Balancer— puede parecer avanzado, pero es exactamente para lo que UserData brilla: tareas
de integración que solo tienen sentido hacerse una vez, en el momento en que la instancia existe
por primera vez, y que serían imposibles de coordinar manualmente a escala.

### Configuración de UserData desde la consola

Para configurar UserData al crear una instancia, ve al final del formulario de creación, en la
sección **Advanced details**. Este apartado de detalles avanzados tiene muchas opciones que
cubriremos en módulos posteriores; por ahora solo nos interesa el campo **User data** al final
de todo.

![Campo User data en Advanced details](/assets/02-compute/04-userdata/userdata-console.png)

En ese campo puedes pegar directamente el contenido de tu script o subir un archivo `.sh`. El
formato es un script de **Bash** — el intérprete de comandos estándar de Linux. Bash es
simplemente el lenguaje en el que escribes instrucciones para la terminal: instalar paquetes con
`yum` o `apt`, crear archivos, iniciar servicios, hacer llamadas a la API de AWS. Si usas
Windows, el equivalente es PowerShell.

El script debe empezar con `#!/bin/bash` en la primera línea — esto le dice al sistema
operativo qué intérprete usar para ejecutarlo.

Aquí un ejemplo real: levantar un servidor web que responda con un mensaje personalizado cuando
alguien accede por HTTP.

```bash
#!/bin/bash
# Instalar Nginx
yum update -y
yum install -y nginx

# Crear página de inicio personalizada
cat > /usr/share/nginx/html/index.html << 'EOF'
<html>
  <body>
    <h1>¡Servidor activo!</h1>
    <p>Esta instancia fue configurada automáticamente con UserData.</p>
  </body>
</html>
EOF

# Iniciar Nginx y habilitarlo para que arranque con el sistema
systemctl start nginx
systemctl enable nginx
```

Con este script, la instancia queda lista y sirviendo tráfico HTTP en cuanto termina de
arrancar — sin que hayas tocado una sola tecla en SSH.

> **Antes de probar:** el Security Group de la instancia debe tener una regla de entrada que
> permita tráfico HTTP en el puerto 80 (TCP, `0.0.0.0/0`). Sin esa regla el servidor Nginx
> responde internamente pero el navegador no puede alcanzarlo. Además, accede siempre por
> `http://` — Nginx no tiene SSL configurado, así que `https://` dará error de conexión rechazada.

Para verificar que el script se ejecutó correctamente, conéctate por SSH y consulta el log de
cloud-init, que captura la salida de cada comando que corrió durante el arranque:

![Conexión SSH y primeras líneas del log de cloud-init](/assets/02-compute/04-userdata/userdata-log-init.png)

Ahí puedes ver el comando `sudo cat /var/log/cloud-init-output.log` y las primeras líneas de
salida. Si el script falló en algún punto, aquí encontrarás el error exacto. Desplazando hacia
abajo en el log aparece la transacción completa de instalación de Nginx — los 7 paquetes
descargados e instalados, incluyendo dependencias, y el mensaje `Complete!` al final:

![Log de instalación de Nginx — transacción completa con "Complete!"](/assets/02-compute/04-userdata/userdata-log-nginx.png)

Con Nginx corriendo, abre un navegador y accede por HTTP a la IP pública de la instancia. La
página personalizada que definimos en el script debe aparecer de inmediato:

![Página web servida por Nginx — resultado del script de UserData](/assets/02-compute/04-userdata/userdata-nginx-web.png)

::: info Detalles importantes de UserData

**Se ejecuta como root.** Todo lo que escribas en el script corre con permisos de superusuario,
así que no necesitas anteponer `sudo` a los comandos.

**Solo se ejecuta en el primer arranque, no en reinicios.** Si detienes y vuelves a iniciar la
instancia, el script no vuelve a correr. Este es el comportamiento por defecto y el que usarás
en la mayoría de los casos.

**Para que se ejecute en cada arranque**, modifica la configuración de cloud-init cambiando la
frecuencia del módulo `scripts-user` a `always`. En Amazon Linux:

```bash
# Agrega esto al final de tu script de UserData:
sed -i 's/^ - scripts-user$/ - [scripts-user, always]/' /etc/cloud/cloud.cfg
```

O directamente en `/etc/cloud/cloud.cfg` cambia `- scripts-user` por `- [scripts-user, always]`.
Útil para scripts de sincronización de configuración que deben ejecutarse cada vez que la
instancia levanta.

**El tamaño máximo es 16 KB.** Si tu configuración es más pesada, lo ideal es que el script
descargue un script más grande desde S3 y lo ejecute.

**Los logs están en `/var/log/cloud-init-output.log`.** Si algo falla en tu UserData, es el
primer lugar donde mirar — tiene la salida completa de cada comando ejecutado.
:::

::: tip ¿UserData se guarda en la AMI?
No exactamente. La AMI captura el **estado del disco** de la instancia — todo lo que UserData
instaló y configuró queda grabado en la imagen — pero el script de UserData en sí **no viaja
con la AMI**. Las instancias que lances desde esa AMI arrancan ya con el software instalado
(porque está en el disco), pero no tienen ningún UserData asignado por defecto y no ejecutan
nada automáticamente.

Esto tiene sentido si lo piensas: el objetivo de crear una AMI desde una instancia ya
configurada es precisamente *no tener que volver a ejecutar el script*. El resultado del script
ya está horneado en el disco. Si quisieras ejecutar algo adicional al arranque de esas nuevas
instancias, le pasas un nuevo UserData distinto al lanzarlas.

Puedes ver o cambiar el UserData de una instancia existente desde la consola en
**Actions → Instance settings → Edit user data** (con la instancia detenida).

**¿Cuándo usar UserData y cuándo una AMI personalizada?**

Usa UserData para configuraciones ligeras, dinámicas o específicas por entorno — variables de
entorno, tokens, nombres de cluster — cosas que cambian entre deploys o entre entornos (dev,
staging, prod). Usa una AMI personalizada cuando la instalación es pesada (compilar
dependencias, descargar runtimes grandes) y quieres que las instancias arranquen rápido sin
repetir ese proceso cada vez. La AMI también te da **versiones precisas**: puedes tener
`ami-app-v1.2.0` con Node 18 y `ami-app-v2.0.0` con Node 20, hacer rollback a una versión
anterior en segundos simplemente cambiando qué AMI usa tu Auto Scaling Group, sin depender de
que un script de UserData reproduzca exactamente el mismo estado.
:::

### Configuración desde CLI

En el comando `aws ec2 run-instances` que ya conocemos, la nueva propiedad es `--user-data`.
Acepta el contenido del script de dos formas:

**Opción 1 — desde un archivo local** (la más común):

```bash
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t3.micro \
  --key-name mi-keypair \
  --security-group-ids sg-0abc123 \
  --user-data file://script.sh
```

El prefijo `file://` le dice a la CLI que lea el contenido del archivo `script.sh` en el
directorio actual y lo envíe como UserData. La CLI se encarga de codificarlo en base64
automáticamente.

**Opción 2 — contenido inline con base64 explícito:**

```bash
aws ec2 run-instances \
  --image-id ami-0abcdef1234567890 \
  --instance-type t3.micro \
  --user-data "$(base64 -w 0 script.sh)"
```

Esta forma es útil en pipelines o scripts donde ya tienes el contenido del script en una
variable y quieres pasarlo directamente sin crear un archivo temporal.

## Metadata en EC2

Mientras una instancia está corriendo, AWS expone un endpoint HTTP que contiene información en
tiempo real sobre esa instancia: su ID, su IP, su región, el rol IAM que tiene asignado. Ese
endpoint es el **Instance Metadata Service (IMDS)**:

```
http://169.254.169.254/latest/meta-data/
```

La IP `169.254.169.254` es una dirección **link-local** — un rango reservado por el estándar de
red que ningún router propaga a internet. AWS la intercepta a nivel de hipervisor: cuando una
instancia hace una petición a esa IP, el tráfico nunca sale de la máquina física que la aloja,
va directamente al servicio de metadata de AWS. Esto significa que **solo el código que corre
dentro de la instancia puede consumir este endpoint** — ni tú desde tu laptop, ni otra
instancia, ni nadie externo puede acceder a él.

¿Cómo se usa esto en la práctica? En un script de UserData puedes llamar al endpoint con `curl`
para obtener la región y construir el nombre de un bucket de S3 sin hardcodearlo. En una
aplicación que corre en la instancia (Node, Python, Java), los SDKs oficiales de AWS llaman
automáticamente a este endpoint para obtener credenciales IAM temporales del rol asignado —
sin que tengas que configurar ninguna clave de acceso en el servidor. Es el mecanismo que hace
posible que una Lambda o una instancia EC2 tenga permisos sin credenciales explícitas en el
código.

### Consultar metadata

Para experimentar con el endpoint conéctate a una instancia por SSH. La forma más directa desde
la consola de AWS es seleccionar la instancia y pulsar **Connect**:

![Lista de instancias con el botón Connect resaltado](/assets/02-compute/04-userdata/metadata-connect-button.png)

Llegarás a la pantalla de conexión, que ofrece cuatro métodos. El más conveniente para pruebas
rápidas es **EC2 Instance Connect** — la primera pestaña:

![Pantalla Connect to instance — pestaña EC2 Instance Connect](/assets/02-compute/04-userdata/metadata-instance-connect.png)

EC2 Instance Connect abre una terminal SSH directamente en el navegador. No te pide tu fichero
`.pem` aunque la instancia tenga un keypair asignado — esto no es una brecha de seguridad: AWS
genera una clave SSH temporal, la inyecta en la instancia durante 60 segundos, establece la
conexión, y la descarta inmediatamente. Tu keypair permanente nunca interviene en este flujo.
El acceso sigue estando controlado por IAM: solo usuarios con permiso `ec2-instance-connect:SendSSHPublicKey`
pueden usar este método.

![Terminal web de EC2 Instance Connect — Amazon Linux 2023 listo](/assets/02-compute/04-userdata/metadata-terminal-connect.png)

Con la terminal abierta, puedes consultar cualquier campo del IMDS directamente con `curl`:

```bash
# Ver todos los campos disponibles (devuelve un listado de rutas)
curl http://169.254.169.254/latest/meta-data/

# Campos más útiles
curl http://169.254.169.254/latest/meta-data/instance-id        # ej: i-0a1b2c3d4e5f
curl http://169.254.169.254/latest/meta-data/public-ipv4        # IP pública
curl http://169.254.169.254/latest/meta-data/local-ipv4         # IP privada
curl http://169.254.169.254/latest/meta-data/placement/region   # región actual
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/  # credenciales IAM
```

### IMDSv1 vs IMDSv2

El servicio tiene dos versiones y la diferencia es enteramente de seguridad.

**IMDSv1** funciona con una petición directa sin ningún tipo de autenticación previa:

```bash
curl http://169.254.169.254/latest/meta-data/instance-id
```

El problema es que cualquier código que corra en la instancia — incluido código malicioso
inyectado — puede leer la metadata sin restricción. Esto lo hace vulnerable a ataques
**SSRF** (_Server-Side Request Forgery_): si tu aplicación web tiene una funcionalidad que
hace peticiones HTTP a URLs que el usuario controla (por ejemplo, un scraper, un webhook, o
una preview de imágenes), un atacante puede pasarle `http://169.254.169.254/latest/meta-data/iam/security-credentials/`
como URL. La aplicación hace la petición desde dentro del servidor, obtiene las credenciales
IAM temporales del rol asignado a la instancia, y se las devuelve al atacante. Con esas
credenciales puede operar en tu cuenta de AWS con todos los permisos de ese rol.

**IMDSv2** cierra esa puerta con un modelo de sesiones en dos pasos:

```bash
# Paso 1: solicitar un token de sesión con tiempo de vida limitado
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 21600")

# Paso 2: usar el token en cada petición de metadata
curl -s http://169.254.169.254/latest/meta-data/instance-id \
  -H "X-aws-ec2-metadata-token: $TOKEN"
```

El token se obtiene con una petición `PUT` — un método HTTP que los navegadores y la mayoría
de las librerías de cliente no emiten automáticamente en flujos SSRF. Además, AWS bloquea el
header `X-Forwarded-For` en las peticiones al IMDS, lo que impide que un proxy o load balancer
interno actúe como intermediario para robar el token. Un atacante SSRF que solo controla la URL
destino no puede completar el paso 1, así que nunca obtiene un token válido.

Puedes verificar si IMDSv2 está forzado en una instancia desde el panel de detalles — busca el
campo **IMDSv2** en la pestaña Details:

![Campo IMDSv2: Required en el panel de detalles de la instancia](/assets/02-compute/04-userdata/metadata-imdsv2-required.png)

::: warning Fuerza IMDSv2 en producción
AWS habilitó IMDSv2 por defecto en instancias nuevas a partir de 2023, pero en instancias
antiguas IMDSv1 puede seguir activo. Para forzar IMDSv2 y deshabilitar IMDSv1 completamente,
puedes hacerlo al crear la instancia en **Advanced details → Metadata accessible → V2 only**,
o con la CLI en una instancia existente:

```bash
aws ec2 modify-instance-metadata-options \
  --instance-id i-0abc123 \
  --http-tokens required \
  --http-endpoint enabled
```
:::

### Metadata en UserData

Un patrón muy común es consultar metadata dentro de los scripts de UserData para que la
configuración sea dinámica sin hardcodear valores por región o instancia:

```bash
#!/bin/bash
# Obtener región e ID de instancia desde metadata (IMDSv2)
TOKEN=$(curl -s -X PUT "http://169.254.169.254/latest/api/token" \
  -H "X-aws-ec2-metadata-token-ttl-seconds: 60")

REGION=$(curl -s http://169.254.169.254/latest/meta-data/placement/region \
  -H "X-aws-ec2-metadata-token: $TOKEN")

INSTANCE_ID=$(curl -s http://169.254.169.254/latest/meta-data/instance-id \
  -H "X-aws-ec2-metadata-token: $TOKEN")

echo "Configurando instancia $INSTANCE_ID en región $REGION"

# Descargar configuración específica de la región desde S3
aws s3 cp s3://mi-bucket-configs/$REGION/app.conf /etc/myapp/app.conf
```

El mismo script funciona en cualquier región sin modificaciones — la instancia sabe dónde está
y ajusta su configuración en consecuencia. Este patrón es la base de cualquier UserData
robusto en entornos multi-región.
