# Amazon EBS: Volúmenes y Snapshots

Hasta ahora hemos hablado de seguridad, gestión y monitoreo, pero no sobre almacenamiento, que
es otro factor fundamental. Como cualquier servidor, nuestras instancias EC2 necesitan un disco
para funcionar: almacenar el sistema operativo, el código de la aplicación, los archivos que
genera durante la ejecución y cualquier otro dato necesario para que el sistema opere.

EC2 ya tiene un disco físico integrado — el **Instance Store** — pero tiene una limitación
crítica: está atado al ciclo de vida de la instancia. Cuando reinicias o eliminas una instancia,
ese disco interno se borra y cualquier dato almacenado en él se pierde irrecuperablemente.

Imagina que tienes un servidor de base de datos con meses de registros de transacciones y, por
un error de configuración, la instancia se termina. Con el disco interno no hay nada que hacer:
los datos desaparecieron junto con la instancia. Para un entorno de producción esto es
inaceptable — necesitamos almacenamiento que sobreviva independientemente del estado de la
instancia.

Para garantizar esa persistencia, AWS ofrece dos servicios de almacenamiento externo: **EBS**
(_Elastic Block Storage_) y **EFS** (_Elastic File System_). Puedes pensarlos como discos duros
externos que conectas al servidor — cuando apagas o eliminas el servidor, el disco externo sigue
ahí intacto.

![Diagrama comparativo: EBS (un bloque) vs EFS (múltiples bloques)](/assets/02-compute/05-ebs/ebs-vs-efs-diagram.png)

La diferencia principal entre los dos es el modelo de conexión:

- **EBS** es un volumen de bloque exclusivo — solo puede estar conectado a una instancia EC2 a
  la vez. Es el equivalente a un disco duro externo USB: mientras está enchufado a tu computador,
  nadie más puede usarlo simultáneamente.
- **EFS** es un sistema de archivos compartido — múltiples instancias pueden montarlo al mismo
  tiempo, como una carpeta de red (NAS) a la que varios equipos acceden en paralelo.

En esta sección nos enfocamos en EBS, que es el tipo de almacenamiento más común en EC2.

---

## EBS

EBS es un servicio de almacenamiento en bloque que AWS reserva y gestiona de forma
independiente de las instancias EC2. "Almacenamiento en bloque" significa que el volumen se
expone como un dispositivo de disco genérico — el sistema operativo de la instancia lo ve
exactamente igual que vería un disco físico interno y puede formatearlo, particionarlo y escribir
en él con las mismas herramientas.

Ese formateo se hace con un **sistema de archivos**: el software que define cómo se organizan y
ubican los datos dentro del disco. En Linux el estándar es **ext4** (_fourth extended
filesystem_), que es el formato con el que Amazon Linux formatea los volúmenes EBS por defecto.
En Windows se usa **NTFS** (_New Technology File System_). Ambos cumplen la misma función —
son el "idioma" que el sistema operativo habla con el disco — pero son incompatibles entre sí:
un volumen formateado en ext4 no lo puede leer Windows directamente, y viceversa.

A estos volúmenes EBS, por convención, se les llama simplemente **volúmenes** — igual que en
Docker. La analogía es directa: un volumen de Docker es almacenamiento que persiste
independientemente del ciclo de vida del contenedor; un volumen EBS persiste independientemente
del ciclo de vida de la instancia. Cuando el contenedor (o la instancia) desaparece, el volumen
sigue existiendo y puede conectarse a otro contenedor (u otra instancia).

### Arquitectura: comunicación por red

Los volúmenes EBS no residen físicamente en el mismo servidor que la instancia EC2 — están en
hardware de almacenamiento separado dentro de la misma Availability Zone, y la comunicación
entre la instancia y el volumen ocurre a través de la red interna de AWS:

![EBS se comunica con EC2 a través de la red interna de AWS](/assets/02-compute/05-ebs/ebs-network-architecture.png)

Esto añade una pequeña latencia que un disco físico local no tendría, pero en casi todos los
casos de uso prácticos la diferencia no es perceptible y el beneficio de la persistencia lo
compensa con creces.

El único escenario donde conviene quedarse con el Instance Store (disco local) es cuando
necesitas latencia extremadamente baja — del orden de microsegundos — y los datos son
completamente efímeros: cachés temporales, buffers de procesamiento, datos que puedes regenerar
en el momento. Para bases de datos, código de aplicación, logs, configuración, o cualquier dato
que no quieras perder, EBS es la respuesta correcta.

### Attach y Detach

Un volumen EBS solo puede estar conectado a una instancia EC2 a la vez, pero esa conexión no es
permanente. Puedes **desconectarlo** (_detach_) de una instancia y **conectarlo** (_attach_) a
otra cuando lo necesites — el volumen conserva todos sus datos intactos durante el proceso.

Esto es útil, por ejemplo, para recuperar datos de una instancia que falló: creas una instancia
nueva, conectas el volumen de la instancia dañada, montas el disco, copias los archivos y listo.
También permite migrar datos entre instancias sin necesidad de transferirlos por red.

## Métricas de rendimiento

El rendimiento de un volumen EBS se describe con dos métricas que miden aspectos distintos del
disco, y entender cuál importa en cada caso de uso es clave para elegir el tipo de volumen
correcto.

### IOPS

**IOPS** (_Input/Output Operations Per Second_) mide cuántas operaciones de lectura o escritura
puede procesar el disco por segundo, sin importar el tamaño de cada operación.

La analogía útil aquí es la de un recepcionista de hotel durante el check-in: IOPS es cuántos
huéspedes puede atender por minuto — cada atención es corta, independiente y requiere acceder a
un registro distinto del sistema. No importa si cada gestión dura 30 segundos o 2 minutos; lo
que limita la cola es cuántas puede resolver por unidad de tiempo. Un disco con alto IOPS puede
saltar rápidamente entre ubicaciones distintas del disco y resolver muchas peticiones pequeñas
y aleatorias en paralelo.

Esto es lo que necesitas cuando tienes operaciones frecuentes, pequeñas e independientes entre
sí — exactamente el patrón de una base de datos SQL con muchas consultas concurrentes (cada
`SELECT` o `INSERT` toca un registro distinto), o un servidor web sirviendo muchas peticiones
simultáneas (cada petición lee archivos distintos del disco). Si el disco tarda en saltar de un
sector a otro, las consultas se acumulan y la aplicación se vuelve lenta aunque cada operación
individual sea pequeña.

### Throughput

**Throughput** (rendimiento) mide cuántos datos puede transferir el disco por segundo,
expresado en MB/s, independientemente de cuántas operaciones separadas conlleve.

La analogía aquí es la de una cinta transportadora en una fábrica: throughput es cuántos
kilogramos de producto puede mover por minuto. No importa si son muchas piezas pequeñas o
pocas piezas grandes — lo que importa es el volumen total de material que avanza. Un disco con
alto throughput puede leer o escribir grandes bloques de datos de forma continua y secuencial
a alta velocidad.

Esto es lo que necesitas cuando procesas archivos grandes de principio a fin — un pipeline de
análisis que lee un CSV de 50 GB, un proceso que renderiza video en 4K, o una copia de seguridad
que vuelca gigabytes de datos al disco de forma secuencial. En estos casos no importa cuántas
operaciones separadas hace el disco por segundo; lo que importa es que el flujo de datos no se
interrumpa.

### Relación entre IOPS y Throughput

Las dos métricas están relacionadas matemáticamente: `Throughput = IOPS × tamaño de bloque`.
Si un disco procesa 1.000 IOPS con bloques de 16 KB, su throughput es 16 MB/s. Si el tamaño
de bloque sube a 256 KB, con los mismos 1.000 IOPS el throughput sube a 256 MB/s.

La analogía con carretera lo hace más intuitivo. Imagina una autopista con casetas de peaje:

- Las **casetas** son los IOPS — cada caseta procesa un vehículo a la vez, y el número de
  casetas determina cuántos vehículos pueden pasar por segundo.
- El **tamaño del vehículo** es el tamaño de bloque — un coche pequeño (16 KB) frente a un
  camión de carga (256 KB).
- El **volumen de mercancía** que cruza la autopista por segundo es el throughput.

Una autopista con muchas casetas (alto IOPS) pero solo coches pequeños (bloques pequeños) da
paso a muchas operaciones por segundo pero mueve poco volumen de datos en total. La misma
autopista con camiones de 18 ruedas (bloques grandes) procesa muchos menos vehículos por
segundo — las casetas se convierten en el cuello de botella — pero cada uno carga tanta
mercancía que el volumen total transferido es enorme.

Por eso cada carga de trabajo satura una dimensión distinta: las bases de datos OLTP son
autopistas llenas de coches pequeños que van y vienen en direcciones aleatorias — lo que se
agota son las casetas (IOPS). El procesamiento de video o analytics es una caravana de
camiones articulados moviéndose en la misma dirección — lo que se agota es el ancho de los
carriles (throughput). Al elegir un tipo de volumen EBS la pregunta no es "¿cuál de las dos?"
sino "¿qué tipo de tráfico va a correr por mi autopista?".

## Creación de Volúmenes y características esenciales

Los volúmenes EBS se gestionan desde la consola de EC2 en la sección **Elastic Block Store →
Volumes** del menú lateral. Al abrirla verás algo parecido a esto:

![Listado de volúmenes EBS en la consola de EC2](/assets/02-compute/05-ebs/ebs-volumes-list.png)

Los cuatro volúmenes que aparecen no fueron creados manualmente — son los discos raíz que AWS
asignó automáticamente al lanzar cada instancia EC2. Cada instancia que creas recibe un volumen
raíz por defecto (el equivalente al `C:\` o al `/` del sistema operativo). En este caso todos
son gp3 de 8 GiB, con 3.000 IOPS y 125 MiB/s de throughput — los valores por defecto de ese
tipo de volumen. La columna **Snapshot ID** indica de qué snapshot partió cada volumen, que en
este caso es la AMI pública de Amazon Linux 2023.

Para crear un volumen adicional pulsa **Create volume**:

![Panel de creación de volumen EBS](/assets/02-compute/05-ebs/ebs-create-volume.png)

Los campos que configuras son los siguientes:

**Volume type** — el tipo de volumen determina el hardware subyacente y el modelo de
rendimiento. Por defecto es gp3, la opción de propósito general más moderna. Los tipos
disponibles los detallamos en la siguiente sección.

**Size (GiB)** — el tamaño del disco. El rango va de 1 GiB hasta 65.536 GiB (64 TiB). Puedes
ampliar un volumen en caliente sin detener la instancia, pero no reducirlo.

**IOPS** — para gp3 el valor base es 3.000 IOPS y puedes subirlo hasta 80.000 de forma
independiente al tamaño, algo que gp2 no permitía. El coste adicional se cobra solo por el
exceso sobre la base.

**Throughput (MiB/s)** — igualmente configurable de forma independiente en gp3, desde 125
hasta 2.000 MiB/s. La pantalla muestra exactamente el formulario de la sección anterior en
la práctica.

**Availability Zone** — crítico: el volumen debe estar en la misma AZ que la instancia a la
que lo vas a conectar. Un volumen en `eu-west-1a` no puede montarse en una instancia de
`eu-west-1b`. Si necesitas moverlo entre AZs, primero creas un snapshot y desde ese snapshot
creas un nuevo volumen en la AZ destino.

**Snapshot ID** — opcional. Si quieres que el volumen parta con datos preexistentes (por
ejemplo, restaurar un backup), selecciona el snapshot aquí. Por defecto se crea vacío.

**Encryption** — puedes cifrar el volumen en reposo con una clave KMS. Una vez creado
cifrado no puede descifrase, y viceversa.

### Desde CLI

```bash
aws ec2 create-volume \
  --volume-type gp3 \
  --size 8 \
  --availability-zone eu-west-1a \
  --iops 3000 \
  --throughput 125
```

De los cinco parámetros del comando, solo `--availability-zone` es estrictamente obligatorio.
`--volume-type` tiene por defecto `gp2` si lo omites, pero siempre conviene especificarlo
explícitamente para no depender del valor por defecto. `--iops` y `--throughput` son opcionales
en gp3 — si los omites se aplican los valores base (3.000 IOPS y 125 MiB/s); son obligatorios
únicamente en io1 e io2, donde no existe un valor por defecto porque el rendimiento se
provisiona explícitamente. `--size` es opcional si creas el volumen desde un snapshot (hereda
el tamaño del snapshot), pero obligatorio si el volumen parte vacío.

El comando devuelve el JSON completo del volumen recién creado con todos sus atributos:

![Resultado del comando create-volume — JSON con VolumeId y State: creating](/assets/02-compute/05-ebs/ebs-create-volume-cli.png)

El campo `State: "creating"` indica que el volumen está siendo aprovisionado. En unos segundos
pasa a `available` y ya puede conectarse a una instancia.

### Características del volumen

Al seleccionar un volumen en la lista, el panel inferior muestra cuatro pestañas. Las tres
primeras son las más relevantes.

**Details** — la ficha técnica completa:

![Pestaña Details de un volumen EBS](/assets/02-compute/05-ebs/ebs-volume-details.png)

Los campos más importantes son **Volume state** (`in-use` cuando está montado en una instancia,
`available` cuando está libre), **Attached resources** (qué instancia lo tiene montado y con
qué nombre de dispositivo — `/dev/xvda` es el disco raíz en Linux), **Availability Zone**, y
**Encryption**. El campo **Source** muestra el snapshot del que partió el volumen, útil para
rastrear el origen de un disco en producción.

**Status checks** — el estado de salud del volumen a nivel de I/O:

![Pestaña Status checks de un volumen EBS](/assets/02-compute/05-ebs/ebs-volume-status.png)

**Status check** en verde significa que el volumen responde correctamente. **I/O status:
Enabled** indica que las operaciones de lectura y escritura están activas. **Auto-enabled I/O**
es una función de resiliencia: si AWS detecta que el volumen tiene datos inconsistentes y lo
deshabilita temporalmente para protegerlos, esta opción lo vuelve a habilitar automáticamente
en lugar de dejarlo bloqueado esperando intervención manual. **Initialization state: Completed**
confirma que el volumen ya está completamente inicializado — los volúmenes creados desde un
snapshot se inicializan de forma lazy (leen el bloque desde S3 la primera vez que se accede),
y este campo indica que ese proceso ya terminó.

**Monitoring** — métricas de CloudWatch en tiempo real:

![Pestaña Monitoring con métricas de CloudWatch](/assets/02-compute/05-ebs/ebs-volume-monitoring.png)

Muestra read/write throughput, IOPS, tamaño medio de operación, longitud de cola y tiempo de
inactividad. Las gráficas aparecen vacías aquí porque el volumen no tiene actividad reciente,
pero en producción son la primera parada para diagnosticar si un cuello de botella viene del
disco o de la red.

### Acciones disponibles

El menú **Actions** agrupa todo lo que puedes hacer con un volumen seleccionado:

![Menú Actions de un volumen EBS](/assets/02-compute/05-ebs/ebs-volume-actions.png)

Las acciones más relevantes son estas:

**Modify volume** — cambia el tipo, el tamaño, los IOPS o el throughput sin detener la
instancia. El cambio se aplica en caliente, aunque puede tardar varios minutos en propagarse.

**Create snapshot** — genera un backup puntual del volumen en S3. Los snapshots son
incrementales: el primero copia todos los bloques, los siguientes solo copian los bloques que
cambiaron desde el anterior. Son la base para mover volúmenes entre AZs, hacer backups y crear
AMIs personalizadas.

**Create snapshot lifecycle policy** — delega la creación de snapshots a un Lifecycle Manager
que los programa automáticamente (diario, semanal, etc.) y gestiona la retención. Imprescindible
en producción para no depender de snapshots manuales.

**Detach volume / Force detach volume** — desconecta el volumen de la instancia. Lo normal es
hacer un `umount` desde dentro de la instancia primero y luego el detach desde la consola. El
force detach lo corta sin esperar al sistema de archivos — útil si la instancia no responde,
pero puede corromper los datos si hay escrituras en curso.

**Delete volume** aparece en gris mientras el volumen está attached — primero hay que
desconectarlo.

## Tipos de Volúmenes en AWS EBS

No todos los volúmenes EBS son iguales. AWS ofrece cinco familias con tecnologías y perfiles de
rendimiento distintos, y elegir la correcta tiene impacto directo tanto en el comportamiento de
tu aplicación como en el costo mensual. La lógica de nomenclatura sigue el mismo patrón que
los tipos de instancia: el número al final indica la generación, así que gp**3** es más moderno
que gp**2**, e io**2** reemplaza a io**1**.

La elección siempre parte de la misma pregunta: ¿qué es lo que más le importa a tu carga de
trabajo — velocidad de operaciones pequeñas (IOPS), volumen de datos transferidos (throughput),
o simplemente almacenamiento barato para datos fríos? La respuesta lleva directamente al tipo
correcto.

::: tip Documentación oficial
Los rangos y límites de cada tipo los mantiene AWS actualizado en su documentación:
[Amazon EBS volume types](https://docs.aws.amazon.com/ebs/latest/userguide/ebs-volume-types.html)
:::

---

### General Purpose SSD — gp3 y gp2

Son los volúmenes de propósito general: buen equilibrio entre rendimiento y precio, y la opción
correcta para la mayoría de las cargas de trabajo. **gp3 es la generación actual y la que debes
usar por defecto**; gp2 sigue disponible por compatibilidad pero gp3 es más barato y más
flexible.

La diferencia clave entre las dos generaciones es cómo se configuran IOPS y throughput:

En **gp2**, ambas métricas están acopladas al tamaño del disco mediante una fórmula fija: 3
IOPS por GiB, con un mínimo de 100 y un techo de 16.000 IOPS. Para conseguir más IOPS en gp2
no tienes otra opción que agrandar el disco aunque no necesites el espacio. El throughput tampoco
es configurable: escala automáticamente hasta 250 MiB/s.

En **gp3**, IOPS y throughput son completamente independientes del tamaño. Puedes tener un
disco de 20 GiB con 10.000 IOPS sin necesidad de sobredimensionar el almacenamiento. Esa
independencia es la razón por la que gp3 es más económico en la mayoría de los casos.

**Rangos de configuración — gp3:**
| Parámetro   | Mínimo       | Máximo        | Incluido en precio base |
|-------------|-------------|---------------|-------------------------|
| Tamaño      | 1 GiB       | 16.384 GiB    | —                       |
| IOPS        | 3.000       | 16.000        | 3.000 IOPS              |
| Throughput  | 125 MiB/s   | 1.000 MiB/s   | 125 MiB/s               |

**Rangos de configuración — gp2:**
| Parámetro   | Mínimo       | Máximo        | Base incluida           |
|-------------|-------------|---------------|-------------------------|
| Tamaño      | 1 GiB       | 16.384 GiB    | —                       |
| IOPS        | 100         | 16.000        | 3 IOPS/GiB (acoplado)   |
| Throughput  | —           | 250 MiB/s     | Automático (no config.)  |

**Cuándo usar gp3:** disco raíz de instancias EC2, entornos de desarrollo, aplicaciones web de
tráfico moderado, bases de datos pequeñas o medianas, cualquier carga donde 16.000 IOPS sea
suficiente. Es el tipo por defecto al crear una instancia y el punto de partida correcto antes
de justificar algo más especializado.

**Cuándo no usar gp3:** si necesitas más de 16.000 IOPS o más de 1.000 MiB/s de throughput de
forma sostenida — ahí entran los volúmenes io.

---

### Provisioned IOPS SSD — io2 e io1

Son los volúmenes de alto rendimiento, diseñados para cargas que necesitan IOPS garantizados,
latencias bajas y consistentes, y en el caso de io2, altísima durabilidad. El adjetivo
"provisioned" significa que AWS reserva el hardware necesario para garantizar el rendimiento
configurado — no hay variaciones ni burst, el disco siempre entrega lo que configuraste.

**io2** es la generación actual. Su diferencial más importante frente a io1 es la durabilidad:
io2 garantiza 99,999% de disponibilidad anual (cinco nueves), frente al 99,8%–99,9% del resto
de tipos. Para bases de datos críticas donde cualquier pérdida de datos es inaceptable, esa
diferencia importa.

io2 también introduce **io2 Block Express**, el tier de mayor rendimiento de toda la familia
EBS: hasta 256.000 IOPS y 4.000 MiB/s en una única instancia Nitro. Está pensado para las
cargas más exigentes: bases de datos Oracle, SAP HANA, o cualquier sistema que mueva terabytes
de datos con baja latencia.

**Rangos de configuración — io2:**
| Parámetro         | Mínimo     | Máximo                           |
|-------------------|-----------|----------------------------------|
| Tamaño            | 4 GiB     | 65.536 GiB                       |
| IOPS              | 100       | 64.000 (256.000 con Block Express)|
| Throughput        | —         | 4.000 MiB/s (Block Express)      |
| Ratio IOPS/GiB    | —         | 500:1                            |

El ratio IOPS/GiB máximo de 500:1 significa que para alcanzar 64.000 IOPS necesitas al menos
128 GiB de disco. Eso evita configuraciones que pidan IOPS imposibles para el tamaño del volumen.

**Rangos de configuración — io1** (referencia, preferir io2):
| Parámetro         | Mínimo     | Máximo        |
|-------------------|-----------|---------------|
| Tamaño            | 4 GiB     | 16.384 GiB    |
| IOPS              | 100       | 64.000        |
| Ratio IOPS/GiB    | —         | 50:1          |

**Cuándo usar io2:** bases de datos relacionales de producción con alta concurrencia (PostgreSQL,
MySQL, Oracle, SQL Server), bases de datos NoSQL de baja latencia (Cassandra, MongoDB en
producción), cualquier sistema que exija latencia sub-milisegundo de forma consistente y donde
el rendimiento degradado tenga impacto directo en el negocio.

**Cuándo no usar io2:** si gp3 con sus 16.000 IOPS máximos te alcanza, el costo de io2 no
está justificado — io2 es significativamente más caro por GiB y por IOPS aprovisionado.

### Throughput Optimized HDD — st1

st1 abandona la tecnología SSD y usa discos magnéticos de alta densidad (HDD), pero optimizados
para throughput: mover grandes volúmenes de datos de forma secuencial a bajo costo. La
contrapartida es que su latencia es mucho mayor que la de cualquier SSD — cada operación tarda
más porque la cabeza lectora del disco físico tiene que desplazarse mecánicamente.

Eso hace que st1 sea completamente inadecuado para accesos aleatorios (una base de datos que
salta entre registros distintos lo encontraría inutilizable), pero excelente para flujos de
datos continuos y predecibles: leer un fichero de 500 GB de principio a fin, procesar logs,
alimentar un pipeline de analytics, o almacenar datos de un data warehouse que se consulta en
batch.

**Rangos de configuración — st1:**
| Parámetro   | Mínimo     | Máximo        |
|-------------|-----------|---------------|
| Tamaño      | 125 GiB   | 16.384 GiB    |
| IOPS        | —         | 500           |
| Throughput  | —         | 500 MiB/s     |

El tamaño mínimo de 125 GiB ya indica para qué está pensado: no es para discos pequeños de
sistema operativo, sino para almacenamiento masivo de datos fríos-pero-accedidos.

**Cuándo usar st1:** almacenamiento de logs a largo plazo, data lakes y repositorios de datos
históricos que se procesan en batch, pipelines de ETL con grandes volúmenes, clusters de
Hadoop o Spark donde el dato se lee secuencialmente. En todos estos casos el throughput importa
y el coste importa — y st1 da hasta 500 MiB/s a una fracción del precio de gp3.

**Cuándo no usar st1:** no puede ser volumen de arranque (solo datos), y es una mala elección
para cualquier carga con accesos aleatorios o que requiera baja latencia.

---

### Cold HDD — sc1

sc1 es el volumen más barato de toda la familia EBS. Usa la misma tecnología HDD que st1 pero
con rendimiento aún más bajo, pensado para datos que rara vez se acceden. Si st1 es para datos
que procesas en batch con cierta frecuencia, sc1 es para datos que simplemente necesitas
guardar y consultar ocasionalmente — archivos históricos, backups secundarios, datos de
compliance que debes retener pero casi nunca lees.

**Rangos de configuración — sc1:**
| Parámetro   | Mínimo     | Máximo        |
|-------------|-----------|---------------|
| Tamaño      | 125 GiB   | 16.384 GiB    |
| IOPS        | —         | 250           |
| Throughput  | —         | 250 MiB/s     |

**Cuándo usar sc1:** archivos históricos que debes retener por normativa pero raramente
consultas, backups de segunda capa, datos de auditoría, cualquier carga donde el coste de
almacenamiento es la prioridad absoluta y el rendimiento es secundario.

**Cuándo no usar sc1:** cualquier carga con accesos frecuentes o que requiera rendimiento
predecible. Para datos fríos con accesos aún más esporádicos y sin necesidad de acceso en
tiempo real, S3 Glacier es una alternativa más barata.

---

### Magnetic — Standard (obsoleto)

La generación original de EBS, de tecnología HDD y rendimiento impredecible. AWS la mantiene
disponible únicamente para compatibilidad con configuraciones antiguas. No tiene límites
definidos de IOPS ni throughput, el rendimiento varía por carga del hardware subyacente, y no
recibe mejoras ni nuevas características.

No uses Magnetic para nada nuevo. Si tienes volúmenes existentes de este tipo, migrarlos a gp3
es inmediato mediante un Modify volume en caliente y, en prácticamente todos los casos,
resultará más barato y con mejor rendimiento.

---

### Resumen comparativo

| Tipo  | Tecnología | IOPS máx.             | Throughput máx. | Uso principal                         |
|-------|-----------|----------------------|-----------------|---------------------------------------|
| gp3   | SSD       | 16.000               | 1.000 MiB/s     | Propósito general — opción por defecto |
| gp2   | SSD       | 16.000               | 250 MiB/s       | Legacy — preferir gp3                 |
| io2   | SSD       | 64.000 / 256.000 BE  | 4.000 MiB/s     | BBDD críticas, alta concurrencia      |
| io1   | SSD       | 64.000               | 1.000 MiB/s     | Legacy — preferir io2                 |
| st1   | HDD       | 500                  | 500 MiB/s       | Big data, logs, acceso secuencial     |
| sc1   | HDD       | 250                  | 250 MiB/s       | Archivos fríos, almacenamiento barato |

::: warning Una restricción importante
Los volúmenes HDD (st1 y sc1) **no pueden usarse como volumen de arranque** de una instancia
EC2. El disco raíz siempre debe ser SSD (gp3, gp2, io1 o io2). Los HDD solo pueden montarse
como volúmenes de datos adicionales.
:::

## Modelo de precios para los volúmenes

El precio de EBS tiene tres componentes independientes que se facturan por separado:

**Almacenamiento (por GiB-mes)** — pagas por el espacio aprovisionado, no por el usado. Si
creas un volumen de 500 GiB y solo usas 50, pagas los 500. La tarifa varía por tipo: gp3 es
más barato que io2, y sc1 es el más barato de todos.

**IOPS aprovisionados** — en gp3, los primeros 3.000 IOPS están incluidos en el precio base
del almacenamiento. Cada IOPS adicional sobre ese umbral tiene un costo por IOPS-mes. En io1 e
io2, todos los IOPS aprovisionados se cobran desde el primero.

**Throughput aprovisionado** — aplica solo a gp3. Los primeros 125 MiB/s están incluidos.
El exceso sobre 125 MiB/s se cobra por MiB/s-mes.

Los snapshots se facturan aparte, por GiB almacenado en S3 (y al ser incrementales, solo pagas
los bloques que cambiaron).

Para estimar el costo de tu configuración concreta antes de crear el volumen, usa la calculadora
oficial de precios de AWS:
[https://aws.amazon.com/ebs/pricing/](https://aws.amazon.com/ebs/pricing/)


## Attach y Detach

Ya definimos attach y detach como el mecanismo de conectar y desconectar volúmenes de una
instancia, pero no hemos visto cómo se hace en la práctica. El attach puede ocurrir en dos
momentos distintos: durante la creación de la instancia, o después, sobre una instancia ya
existente.

### Configurar volúmenes al crear la instancia

En el panel de creación de EC2, la sección **Configure storage** muestra el volumen raíz que
se asignará automáticamente:

![Panel Configure storage en creación de instancia EC2](/assets/02-compute/05-ebs/ebs-attach-storage-config.png)

El volumen que aparece por defecto es el **Root Volume** — el disco raíz donde vive el sistema
operativo. Es el equivalente al disco C:\ en Windows o al punto de montaje `/` en Linux: sin
él la instancia no puede arrancar. Por eso no tiene botón de Remove y no puede eliminarse
mientras la instancia exista.

No todos los tipos de volumen pueden ser el volumen raíz. Solo los SSD (gp3, gp2, io1, io2)
son válidos como disco de arranque — los HDD (st1, sc1) únicamente pueden usarse como volúmenes
de datos adicionales, nunca como raíz.

La sección **File systems** al pie del panel ofrece conectar directamente un sistema de
archivos compartido a la instancia en el momento del lanzamiento: **EFS** (sistema de archivos
NFS gestionado por AWS), **FSx** (sistemas de archivos de alto rendimiento como Windows File
Server o Lustre), o **S3 Files** (acceso a S3 como sistema de archivos, en preview). Seleccionar
`None` es lo habitual cuando solo necesitas el volumen EBS de bloque.

Puedes añadir cuantos volúmenes adicionales necesites con **Add new volume**:

![Tres volúmenes adicionales añadidos — el raíz sin botón Remove, los demás con él](/assets/02-compute/05-ebs/ebs-attach-multiple-volumes.png)

Todos los volúmenes que añades aquí se crean y se conectan automáticamente a la instancia en
el momento del lanzamiento. El volumen raíz no tiene botón **Remove**; los volúmenes adicionales
sí pueden eliminarse del listado antes de lanzar.

### Vista avanzada de almacenamiento

La vista simple solo muestra tamaño y tipo. Para acceder a todos los parámetros de cada
volumen, pulsa **Advanced** en la esquina superior derecha:

![Enlace Advanced en el panel de Configure storage](/assets/02-compute/05-ebs/ebs-attach-advanced-link.png)

![Vista avanzada con configuración completa por volumen](/assets/02-compute/05-ebs/ebs-attach-advanced-config.png)

Aquí puedes configurar cada volumen de forma independiente. Los campos más relevantes son:

**Device name** — el nombre con el que el sistema operativo de la instancia identificará este
disco. El volumen raíz siempre usa `/dev/xvda`. Para volúmenes adicionales puedes elegir entre
`/dev/sdb`, `/dev/sdc`, etc. Al pulsar el selector se despliega la lista completa:

![Selector de Device name — /dev/sdb seleccionado, /dev/sdc y /dev/sdd en gris](/assets/02-compute/05-ebs/ebs-attach-device-names.png)

Las opciones en gris (`/dev/sdc`, `/dev/sdd`) ya están ocupadas por otros volúmenes del mismo
formulario. El sufijo importa: `a` está reservado para el disco raíz, y `b` en adelante son
para datos. En la práctica, en instancias Nitro de AWS el nombre que eliges (`/dev/sdb`) se
convierte en un symlink al dispositivo NVMe real (`/dev/nvme1n1`), pero esto lo veremos en
detalle en la sección de formateo.

**Delete on termination** — define qué ocurre con el volumen cuando se elimina la instancia.
El volumen raíz viene con `Yes` por defecto (se elimina con la instancia), lo cual tiene
sentido: el disco del SO no necesitas conservarlo. Los volúmenes de datos adicionales vienen
con `No` por defecto, de forma que los datos persisten aunque elimines la instancia. Este
comportamiento es el correcto para la mayoría de los casos, pero es importante revisarlo: un
volumen huérfano que nadie eliminó explícitamente sigue generando costo mensual aunque ninguna
instancia lo esté usando.

**Snapshot** — el snapshot del que parte el volumen. El raíz parte del snapshot de la AMI;
los volúmenes adicionales parten de ningún snapshot (vacíos) a menos que especifiques uno.

### Attach desde CLI

Si el volumen ya existe y quieres conectarlo a una instancia en marcha, el comando es
`aws ec2 attach-volume`:

```bash
aws ec2 attach-volume \
  --volume-id vol-0a2633be63445e362  \ 
  --instance-id i-0cd4741c1c9ff6a94   \
  --device /dev/xvdf
```

![Resultado del comando attach-volume — State: attaching](/assets/02-compute/05-ebs/ebs-attach-cli-result.png)

El parámetro `--device` es el nombre con el que el volumen aparecerá en el sistema operativo.
La convención es empezar por `/dev/xvdf` para el primer volumen adicional y continuar con
`xvdg`, `xvdh`, etc. Evita `/dev/xvda` (disco raíz) y `/dev/xvdb` (reservado para swap en
algunas AMIs). En instancias Nitro, AWS mapea `/dev/xvdf` a `/dev/nvme1n1` mediante un symlink
— el nombre que eliges sigue siendo válido y reconocible, pero el dispositivo físico real que
verás dentro de la instancia es el NVMe.

El campo `State: "attaching"` en la respuesta indica que el proceso está en curso. En unos
segundos el estado pasará a `attached`.

### Verificar desde la consola

Para confirmar que el volumen quedó correctamente conectado, ve a la pestaña **Storage** de
la instancia:

![Pestaña Storage de la instancia — /dev/xvda root y /dev/xvdf recién attached](/assets/02-compute/05-ebs/ebs-attach-console-verify.png)

Puedes ver ambos volúmenes en **Block devices**: el raíz en `/dev/xvda` y el nuevo en
`/dev/xvdf`, ambos con estado `In-use` y `Attached`. El campo **Root device name** en la
sección superior te dice cuál de los dos es el disco de arranque:

![Root device name: /dev/xvda destacado](/assets/02-compute/05-ebs/ebs-attach-root-device.png)

## Formatear volúmenes

Un volumen EBS recién creado y conectado a una instancia es un dispositivo de bloque en bruto —
existe como disco reconocido por el sistema operativo, pero está completamente vacío. Antes de
poder guardar archivos en é necesitas **formatearlo**: escribir en él las estructuras de
metadatos que definen cómo se van a organizar y localizar los datos. Esas estructuras son el
sistema de archivos, y el proceso de escribirlas se llama formateo.

Sin formateo, el sistema operativo no sabe leer ni escribir en el disco — es como tener una
hoja en blanco sin ningún sistema de índices. El formateo crea ese sistema de índices (inodos,
tablas de bloques, journal de recuperación) que permite que comandos como `ls`, `cp` o `write`
funcionen sobre él.

El formato no se puede establecer al crear el volumen desde la consola de AWS — EBS solo
provisiona el hardware de almacenamiento. El formateo es una operación del sistema operativo
que se hace después, dentro de la instancia.

### Ver los discos disponibles

El primer paso siempre es listar los dispositivos de bloque que reconoce el sistema operativo:

```bash
lsblk
```

![Salida de lsblk — nvme0n1 con particiones montadas, nvme1n1 sin montar](/assets/02-compute/05-ebs/ebs-format-lsblk.png)

`lsblk` es una utilidad nativa de Linux que lista todos los dispositivos de bloque. La salida
muestra dos discos: `nvme0n1` (el volumen raíz) con sus tres particiones ya montadas (`/`,
`/boot/efi`), y `nvme1n1` (el volumen que acabamos de conectar) sin ninguna partición ni
punto de montaje — está ahí, pero vacío.

### Verificar que no tiene sistema de archivos

Antes de formatear conviene confirmar que el disco realmente no tiene un sistema de archivos
previo (para no borrar datos por error si fuera un volumen restaurado desde snapshot):

```bash
file -s /dev/xvdf
sudo file -s /dev/nvme1n1
```

![file -s: /dev/xvdf es symlink a nvme1n1; /dev/nvme1n1 devuelve "data"](/assets/02-compute/05-ebs/ebs-format-file-check.png)

La primera línea confirma que `/dev/xvdf` es un **symbolic link** a `/dev/nvme1n1`. En
instancias Nitro, AWS expone los discos EBS como dispositivos NVMe (`nvme*`) y crea symlinks
con los nombres convencionales (`xvdf`, `xvdg`...) para mantener compatibilidad. El dispositivo
real es el NVMe.

La segunda línea es la clave: `sudo file -s /dev/nvme1n1` devuelve `data`. Eso significa que
el comando no reconoció ninguna firma de sistema de archivos conocida — el disco contiene bytes
sin estructura, es decir, está sin formatear. Si tuviera un sistema de archivos ext4, la
respuesta sería algo como `Linux rev 1.0 ext4 filesystem data`.

Para confirmar qué volumen EBS de AWS corresponde a este dispositivo NVMe puedes usar:

```bash
sudo ebsnvme-id /dev/nvme1n1
```

![ebsnvme-id muestra el Volume ID y el device name](/assets/02-compute/05-ebs/ebs-format-ebsnvme-id.png)

`ebsnvme-id` es una herramienta incluida en Amazon Linux que lee los metadatos del controlador
NVMe y te dice exactamente qué volumen EBS está detrás de ese dispositivo — útil cuando tienes
varios volúmenes conectados y necesitas confirmar cuál es cuál antes de operar sobre él.

### Aplicar el formato ext4

Con la identidad del disco confirmada, aplicamos el sistema de archivos:

```bash
sudo mkfs -t ext4 /dev/nvme1n1
```

![Salida de mkfs ext4 — bloques, inodos, UUID y journal creados](/assets/02-compute/05-ebs/ebs-format-mkfs.png)

`mkfs` significa _make filesystem_. El flag `-t ext4` especifica el tipo — ext4 es el sistema
de archivos estándar de Linux moderno. La salida detalla lo que se escribió en el disco:

- **2097152 blocks de 4k** — el disco se divide en bloques de 4 KiB, que es la unidad mínima
  de almacenamiento. Cada lectura o escritura opera sobre bloques completos.
- **524288 inodos** — un inodo es la estructura que almacena los metadatos de un archivo
  (permisos, propietario, fechas, punteros a los bloques de datos). Hay un inodo por archivo
  posible; si se agotan los inodos no puedes crear más archivos aunque quede espacio en disco.
- **UUID** — identificador único del sistema de archivos, usado para montaje persistente en
  `/etc/fstab`.
- **Superblocks** — copias de los metadatos críticos del sistema de archivos distribuidas en
  distintas ubicaciones del disco. Si el superblock principal se corrompe, ext4 puede recuperarse
  usando una de las copias de seguridad.
- **Journal** — registro de transacciones que permite a ext4 recuperar el estado consistente
  del sistema de archivos tras un fallo inesperado (corte de luz, crash del kernel) sin tener
  que escanear el disco completo.

### Montar el volumen

Formatear el disco solo escribe las estructuras. Para que el sistema operativo pueda leer y
escribir archivos en él necesitas **montarlo** — es decir, conectar ese dispositivo a un
directorio del árbol de archivos de la instancia. A partir de ese momento, cualquier operación
sobre ese directorio opera sobre el disco EBS.

```bash
mkdir volumen
sudo mount /dev/nvme1n1 volumen
```

![Mount, ls, y df -h mostrando el volumen montado en /home/ec2-user/volumen](/assets/02-compute/05-ebs/ebs-format-mount.png)

Primero creamos el directorio que servirá como punto de montaje (`mkdir volumen`), luego
montamos el dispositivo sobre él (`sudo mount`). A partir de ese instante el directorio
`volumen/` es la puerta de entrada al disco EBS.

`ls` dentro del directorio recién montado muestra `lost+found` — un directorio especial que
ext4 crea automáticamente durante el formateo. Lo usa el comando `fsck` para depositar archivos
rescatados en una recuperación de disco; en condiciones normales está vacío y puedes ignorarlo.

`df -h volumen/` confirma el estado del montaje: el dispositivo `/dev/nvme1n1` tiene 7,8 GiB
totales, 24 KiB usados (las estructuras del propio sistema de archivos) y 7,4 GiB disponibles,
montado en `/home/ec2-user/volumen`.

::: warning El montaje no persiste al reiniciar
`sudo mount` es temporal — si la instancia se reinicia el volumen queda conectado (attached)
pero no montado. Para que se monte automáticamente en cada arranque debes añadir una entrada
en `/etc/fstab` usando el UUID del volumen:

```bash
# Obtener el UUID
sudo blkid /dev/nvme1n1

# Añadir al final de /etc/fstab:
UUID=c3958c56-2961-43e8-a337-9f3cdd1134a5  /home/ec2-user/volumen  ext4  defaults,nofail  0  2
```

El flag `nofail` es importante: evita que la instancia quede bloqueada en el arranque si por
algún motivo el volumen EBS no está disponible.
:::

La forma recomendada de añadir la entrada a fstab es usar el **UUID** del volumen en lugar del
nombre de dispositivo (`/dev/nvme1n1`), porque el UUID es permanente — el nombre puede cambiar
entre reinicios cuando hay múltiples volúmenes. El proceso en dos comandos:

![Obtener UUID con blkid y añadir la entrada a fstab con tee](/assets/02-compute/05-ebs/ebs-fstab-uuid-command.png)

```bash
# Paso 1: obtener el UUID del volumen
sudo blkid /dev/nvme1n1

# Paso 2: escribir la entrada en /etc/fstab (sustituye el UUID por el tuyo)
echo 'UUID=<tu-uuid>  /volumen  ext4  defaults,nofail  0  2' | sudo tee -a /etc/fstab
```

`tee -a` escribe en el archivo sin sobrescribir las líneas existentes, y muestra el resultado
en pantalla para confirmar que quedó correcto. Para verificar, abre el archivo directamente:

```bash
vim /etc/fstab
```

![Contenido del archivo /etc/fstab con la nueva entrada del volumen al final](/assets/02-compute/05-ebs/ebs-fstab-file.png)

Verás las entradas del sistema (volumen raíz en `xfs`, partición EFI en `vfat`) seguidas de
la línea que acabas de añadir con tu UUID, punto de montaje, `ext4` y `defaults,nofail 0 2`.
Para validar sin reiniciar ejecuta `sudo mount -a` — si no hay error, la configuración es correcta.

### Attach desde la consola de la instancia

Hasta ahora hicimos el attach desde el panel **Elastic Block Store → Volumes**. Hay una ruta
equivalente: seleccionar la instancia en EC2 → **Actions → Storage → Attach volume**:

![Menú Actions → Storage → Attach volume desde la lista de instancias EC2](/assets/02-compute/05-ebs/ebs-attach-from-instance-menu.png)

Este flujo abre el mismo formulario pero pre-poblado con el ID y la Availability Zone de la
instancia. Desde aquí seleccionas el volumen (debe estar en la misma AZ) y el nombre de dispositivo:

![Formulario Attach volume iniciado desde la instancia](/assets/02-compute/05-ebs/ebs-attach-from-instance-form.png)

::: info ¿Este attach monta el volumen de forma persistente?
No. El attach — desde el panel de volúmenes o desde la consola de instancias — **solo conecta
el dispositivo de bloque al sistema**. Es equivalente a enchufar un disco: el kernel lo
reconoce y aparece en `lsblk`, pero no está montado en ningún punto del sistema de archivos.

Para usarlo debes seguir los mismos dos pasos que ya vimos:
1. `sudo mount /dev/nvme1n1 /volumen` — montaje temporal (se pierde al reiniciar)
2. Entrada en `/etc/fstab` con el UUID — montaje automático en cada arranque

El attach sí es persistente en otro sentido: si detienes y reinicias la instancia, el volumen
permanece adjunto (no se desconecta), pero sin la entrada en fstab tendrás que montarlo
manualmente cada vez que la instancia arranque.
:::

## Modificar Volúmenes

Una de las ventajas de EBS frente al almacenamiento local es que puedes cambiar las
características de un volumen **sin detener la instancia**. Tamaño, tipo, IOPS y throughput
son todos modificables en caliente — la instancia sigue operando mientras el volumen se
reconfigura en segundo plano.

Para modificar desde la consola, selecciona el volumen en el panel de EBS y abre
**Actions → Modify volume**:

![Menú Actions → Modify volume en el panel de volúmenes EBS](/assets/02-compute/05-ebs/ebs-modify-volume-menu.png)

El formulario expone todos los parámetros ajustables:

![Formulario Modify volume — type, size, IOPS y throughput configurables](/assets/02-compute/05-ebs/ebs-modify-volume-form.png)

Puedes cambiar cualquier combinación en una sola operación:

- **Volume type**: migrar entre gp3, gp2, io2, io1, st1 o sc1 sin desmontar
- **Size**: solo puede aumentar, nunca reducir. El volumen crece mientras está en uso
- **IOPS**: en gp3 son independientes del tamaño, ajustables entre 3.000 y 80.000
- **Throughput**: también independiente en gp3, de 125 a 2.000 MiB/s

::: warning El sistema de archivos no crece automáticamente con el volumen
EBS agranda el dispositivo de bloque, pero el sistema de archivos dentro sigue ocupando el
tamaño original. Para aprovechar el espacio añadido debes expandirlo explícitamente:

```bash
# Para ext4:
sudo resize2fs /dev/nvme1n1

# Para xfs:
sudo xfs_growfs /volumen
```

Ambos comandos funcionan con el volumen montado — no necesitas desmontarlo ni detener la instancia.
:::

### Modificar desde la CLI

```bash
aws ec2 modify-volume \
  --volume-id vol-0123456789abcdef0 \
  --size 100 \
  --volume-type gp3 \
  --iops 5000 \
  --throughput 250
```


Puedes especificar cualquier subconjunto de parámetros — solo tamaño, solo IOPS, o todos a la
vez. La modificación es asíncrona: el volumen pasa al estado `modifying` y vuelve a `in-use`
cuando termina. Para monitorear el progreso:

```bash
aws ec2 describe-volumes-modifications --volume-id vol-0123456789abcdef0
```

## Encriptación de EBS

EBS soporta encriptación en reposo con AES-256 usando claves gestionadas por AWS KMS. La
encriptación cubre los datos almacenados en el volumen, los snapshots generados desde él, los
datos en tránsito entre el volumen y la instancia dentro de la red interna de AWS, y los
volúmenes nuevos creados a partir de esos snapshots.

Puedes habilitar la encriptación **por defecto a nivel de cuenta**: todos los volúmenes nuevos
en esa región quedarán encriptados automáticamente sin configuración adicional por volumen:


```bash
# Verificar el estado actual
aws ec2 get-ebs-encryption-by-default
# { "EbsEncryptionByDefault": false }

# Habilitar encriptación por defecto para todos los volúmenes nuevos en la región
aws ec2 enable-ebs-encryption-by-default
# { "EbsEncryptionByDefault": true }
```

::: info ¿Por qué importa la encriptación por defecto?

**Cumplimiento normativo**: PCI-DSS, HIPAA, SOC 2 y la mayoría de marcos de seguridad exigen
datos en reposo encriptados. Habilitar esta opción cubre ese requisito sin gestión manual
por recurso.

**Protección de snapshots**: un snapshot EBS puede compartirse entre cuentas de AWS o hacerse
público. Si el volumen original no estaba encriptado, el snapshot expone todos los datos en
claro. Con encriptación por defecto los snapshots heredan la encriptación y solo son utilizables
por cuentas con acceso a la clave KMS correspondiente.

**Sin impacto en rendimiento**: la encriptación y desencriptación ocurren en el hardware del
hipervisor, fuera del sistema operativo de la instancia. No consumes CPU de tu EC2 ni hay
latencia perceptible.

La clave por defecto es `aws/ebs` (administrada por AWS). Para control granular — rotación
propia, auditoría separada, revocación de acceso independiente — puedes especificar una
Customer Managed Key (CMK) al crear cada volumen.
:::

## Snapshots

Un snapshot es una copia de seguridad de un volumen EBS capturada en un momento específico
del tiempo. Son el equivalente a los backups de un ordenador o una base de datos: su objetivo
es evitar pérdidas de información y facilitar la recuperación de datos. No son la única capa
de protección que AWS ofrece, pero sí una de las más importantes y directas de gestionar.

Por detrás, los snapshots se almacenan en S3 — no en el S3 de tu cuenta, sino en el
almacenamiento interno gestionado por AWS. Esto les da la durabilidad de S3 (99,999999999%,
once nueves), replicación automática entre múltiples zonas de disponibilidad, y acceso
independiente del estado de la instancia o el volumen de origen.

**Los snapshots son incrementales.** El primero captura el volumen completo. Cada snapshot
posterior solo almacena los bloques que cambiaron desde el último. Ejemplo concreto: tienes
un volumen de 8 GiB con 5 GiB de datos. El primer snapshot ocupa ≈5 GiB. En los siguientes
días escribes 1 GiB nuevo de datos. El segundo snapshot solo ocupa ≈1 GiB — no vuelve a
copiar los 5 GiB que no cambiaron. El almacenamiento total es ≈6 GiB en lugar de ≈10 GiB.
AWS factura solo el almacenamiento real, por lo que los snapshots frecuentes son más baratos
de lo que parecen.

### Crear Snapshots

El panel de Snapshots vive en **Elastic Block Store → Snapshots** en la barra lateral de EC2.
Desde ahí puedes ver todos los snapshots existentes y crear nuevos con el botón
**Create snapshot**:

![Panel Elastic Block Store → Snapshots con botón Create snapshot resaltado](/assets/02-compute/05-ebs/ebs-snapshots-list.png)

Al crear un snapshot tienes dos modos según el campo **Resource type**:

**Volume** — captura un volumen específico. Seleccionas el volumen por ID. Útil para backups
puntuales de un disco de datos o el volumen raíz de una instancia:

![Formulario Create snapshot con Resource type = Volume](/assets/02-compute/05-ebs/ebs-snapshot-create-form.png)

**Instance** — captura todos los volúmenes adjuntos a una instancia de forma coordinada.
En lugar de un Volume ID seleccionas un Instance ID. AWS toma los snapshots de todos los
volúmenes de esa instancia simultáneamente, garantizando consistencia entre ellos — esencial
si tienes, por ejemplo, una base de datos con datos y logs en volúmenes separados:

![Formulario Create snapshot con Resource type = Instance](/assets/02-compute/05-ebs/ebs-snapshot-create-instance.png)

Ambos modos aceptan una descripción opcional (255 caracteres) y tags para identificar y
organizar los snapshots en entornos con muchos volúmenes.

Una vez creado, el snapshot aparece en la lista con estado **Pending** mientras se completa
la copia, y pasa a **Completed** cuando termina. El panel de detalles muestra el tamaño real
del snapshot (que puede ser menor que el volumen por la naturaleza incremental), el progreso,
el volumen de origen, y si está encriptado:

![Detalle del snapshot recién creado — estado Pending, size del volumen, sección Encryption](/assets/02-compute/05-ebs/ebs-snapshot-detail.png)

### Creación Automática

Crear snapshots manualmente no escala. Para automatizarlo AWS ofrece **Amazon Data Lifecycle
Manager (DLM)**, que permite definir políticas que ejecutan snapshots de forma recurrente y
gestionan su retención automáticamente.

La forma más rápida de acceder es desde el panel de Volúmenes: selecciona un volumen y abre
**Actions → Create snapshot lifecycle policy**:

![Menú Actions → Create snapshot lifecycle policy desde el panel de Volúmenes EBS](/assets/02-compute/05-ebs/ebs-snapshot-lifecycle-menu.png)

El asistente tiene tres pasos.

**Paso 1 — Specify settings:** defines qué recursos va a proteger la política y con qué rol:

![Paso 1: Target resources, Target resource tags, Description e IAM role](/assets/02-compute/05-ebs/ebs-dlm-step1-settings.png)

Los campos clave son **Target resource types** (Volume o Instance) y **Target resource tags**.
Este último punto es crítico: DLM no selecciona recursos por nombre ni por ID, sino por tag.
La política buscará todos los volúmenes (o instancias) que tengan el tag que especifiques aquí
y les aplicará el ciclo de vida automáticamente. Sin un tag asociado al volumen, la política no
puede identificarlo y nunca lo incluirá en los backups. Por eso antes de activar una política
DLM debes asegurarte de que los volúmenes que quieres proteger tienen el tag configurado.

Al final del paso 1 también defines si la política se activa al crear (**Enabled**) o la
guardas en estado inactivo para activarla más tarde:

![Tags opcionales de la política y Policy status (Enabled / Not enabled)](/assets/02-compute/05-ebs/ebs-dlm-step1-policy-status.png)

**Paso 2 — Configure schedule:** define la cadencia de los snapshots y cuánto tiempo se conservan:

![Paso 2: Schedule name, Frequency, Every, Starting at y Retention type](/assets/02-compute/05-ebs/ebs-dlm-step2-schedule.png)

Los parámetros más importantes son la **frecuencia** (Daily, Weekly, Monthly, o custom) y el
**tipo de retención**: por **Count** (conserva los N snapshots más recientes, borra el más
antiguo cuando se supera el límite) o por **Age** (conserva snapshots durante X días y los
borra pasado ese tiempo):

![Dropdown Retention type: Count vs Age](/assets/02-compute/05-ebs/ebs-dlm-retention-type.png)

El paso 2 también expone opciones avanzadas de la política, que por defecto están colapsadas:

![Advanced settings: Tagging, Snapshot archiving, Fast snapshot restore, Cross-Region copy, Cross-account sharing](/assets/02-compute/05-ebs/ebs-dlm-step2-advanced.png)

Las más relevantes para arquitecturas más avanzadas son **Cross-Region copy** (replica cada
snapshot automáticamente en otra región como parte de una estrategia de disaster recovery) y
**Cross-account sharing** (comparte los snapshots con otras cuentas AWS de tu organización).

**Paso 3 — Review and create:** resumen completo antes de confirmar:

![Revisión final de la política con settings y schedule configurados](/assets/02-compute/05-ebs/ebs-dlm-review.png)

### Creación desde CLI

Puedes crear snapshots y políticas DLM desde la CLI sin pasar por la consola:

```bash
# Snapshot de un volumen específico
aws ec2 create-snapshot \
  --volume-id vol-0123456789abcdef0 \
  --description "Backup manual - servidor de produccion"

# Snapshot multi-volumen de una instancia completa (consistente entre volúmenes)
aws ec2 create-snapshots \
  --instance-specification InstanceId=i-0abc123,ExcludeBootVolume=false \
  --description "Backup completo instancia app-server"

# Listar snapshots propios con filtro por estado
aws ec2 describe-snapshots \
  --owner-ids self \
  --filters Name=status,Values=completed
```

`create-snapshot` es síncrono en el envío pero el proceso de copia es asíncrono: el comando
responde inmediatamente con el ID del snapshot y el estado `pending`. Usa
`describe-snapshots --snapshot-ids snap-xxx` para monitorear el progreso hasta que el estado
cambie a `completed`.

### Funcionalidades

Los snapshots no son solo backups pasivos. Desde el menú **Actions** de un snapshot puedes
desencadenar varias operaciones:

![Menú Actions de un snapshot — Create volume from snapshot, Create image from snapshot, Copy snapshot](/assets/02-compute/05-ebs/ebs-snapshot-actions.png)

**Create volume from snapshot**: restaura el contenido del snapshot en un volumen EBS nuevo.
Puedes cambiarlo de tipo (migrar de gp2 a gp3), de tamaño (solo agrandar), de zona de
disponibilidad, o de región. Es la operación base para restaurar datos o mover volúmenes
entre zonas.

**Create image from snapshot**: registra el snapshot como una AMI. Sí — cuando ves "image"
aquí se refiere a una Amazon Machine Image. Si el snapshot es del volumen raíz de una
instancia, puedes convertirlo directamente en una AMI desde la que lanzar instancias
idénticas. Es uno de los flujos más usados para crear AMIs personalizadas: configuras una
instancia, haces snapshot del raíz, y registras ese snapshot como AMI.

**Copy snapshot — cross-region DR**: permite copiar un snapshot a otra región de AWS. Este es
el mecanismo fundamental para disaster recovery: si la región `eu-west-1` queda inaccesible,
los datos copiados a `us-east-1` permiten restaurar los volúmenes allí y recuperar el servicio.

![Menú Actions → Copy snapshot seleccionado](/assets/02-compute/05-ebs/ebs-snapshot-copy-menu.png)

El formulario de copia permite elegir la región destino, añadir una descripción para
identificar la copia, especificar si debe ir encriptada, y habilitar una copia con tiempo
garantizado (**Time-based copy**) si necesitas cumplir un SLA de RPO definido:

![Formulario Copy snapshot — Source snapshot, Destination Region, Time-based copy, Encryption](/assets/02-compute/05-ebs/ebs-snapshot-copy-form.png)

Desde CLI:

```bash
aws ec2 copy-snapshot \
  --source-region eu-west-1 \
  --source-snapshot-id snap-0abcdef1234567890 \
  --destination-region us-east-1 \
  --description "DR copy - backup produccion"
```

El comando se ejecuta desde la región **destino** (o especificándola con `--destination-region`)
y requiere que la región origen esté disponible para iniciar la transferencia. Al igual que
`create-snapshot`, la copia es asíncrona: obtienes el ID del snapshot de destino y monitoreas
con `describe-snapshots` hasta que llega a `completed`.
