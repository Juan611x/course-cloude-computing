# Seguridad en EC2

Lanzar una instancia es solo la mitad del trabajo. La otra mitad — la que determina si esa
instancia puede ser explotada, comprometida o simplemente inaccesible cuando más la necesitas —
es la seguridad. En el mundo cloud, la superficie de ataque es inherentemente pública: tus
servidores tienen IPs accesibles desde Internet, y sin las configuraciones correctas, cualquier
actor malicioso del planeta puede intentar llegar a ellos.

AWS no te lanza desprotegido. Alrededor de EC2 existe un conjunto de herramientas de seguridad
que, bien configuradas, forman capas de defensa complementarias. En esta sección cubrimos las
más fundamentales para operar instancias de forma segura: los **Security Groups**, que controlan
qué tráfico puede entrar y salir de tus instancias, y los **Key Pairs**, que garantizan que solo
tú puedas conectarte a ellas.

---

## Security Groups

Imagina que acabas de lanzar un servidor EC2 con una aplicación web. El servidor está corriendo,
tiene una IP pública y técnicamente es accesible desde cualquier punto de Internet. ¿Qué impide
que alguien intente conectarse al puerto de tu base de datos? ¿O que un bot escanee tu servidor
buscando puertos abiertos vulnerables? Sin ningún mecanismo de filtrado, absolutamente nada.

Ahí es donde entran los **Security Groups** (SGs). No son un servicio independiente ni un
componente que instalas — son una **configuración de firewall virtual** que AWS aplica a nivel
de red, antes de que cualquier paquete llegue siquiera a tu instancia. Cada regla que defines
en un SG le dice a AWS qué tráfico dejar pasar y qué tráfico descartar silenciosamente.

![Concepto de Security Group](/assets/02-compute/02-security/sg-concept.png)

Visualmente, el Security Group actúa como un perímetro alrededor de la instancia: el tráfico
entrante (*inbound*) debe cruzar ese perímetro para llegar al servidor, y el tráfico saliente
(*outbound*) debe cruzarlo en sentido inverso para salir. El SG evalúa cada conexión contra
sus reglas y decide si la permite o la bloquea.

**Tráfico entrante (Inbound)** es cualquier comunicación que llega *hacia* tu instancia desde
el exterior: un usuario abriendo tu sitio web en el navegador, un desarrollador conectándose
por SSH, otro servicio interno consultando tu API. Cada una de estas conexiones entra por un
puerto específico.

**Tráfico saliente (Outbound)** es lo que tu instancia envía *hacia* afuera: las respuestas
a las peticiones recibidas, llamadas a APIs externas, descarga de actualizaciones, conexiones
a bases de datos en otra instancia.

### Reglas del juego

Toda instancia EC2 **debe tener al menos un Security Group** asignado — no es opcional. Si no
asignas uno explícitamente al lanzarla, AWS le asigna automáticamente el SG por defecto de la
VPC. Además, una instancia puede tener hasta **5 Security Groups** simultáneamente, y las reglas
de todos se suman: si cualquiera de los SGs permite el tráfico, pasa.

Al lanzar una instancia, el asistente de configuración ya te presenta la sección de Security
Groups con opciones básicas preconfiguradas:

![Security Group en el asistente de lanzamiento](/assets/02-compute/02-security/sg-launch-wizard.png)

Por defecto el asistente crea un SG nuevo llamado `launch-wizard-N` con una sola regla de
entrada: SSH (puerto 22) desde cualquier IP (`0.0.0.0/0`). Esto permite conectarte desde
donde estés, pero es una configuración abierta que deberías restringir a tu IP en entornos
de producción. Las opciones de HTTP y HTTPS vienen desmarcadas — las activas solo si tu
instancia va a servir tráfico web.

::: warning Nunca dejes SSH abierto a todo el mundo en producción
`0.0.0.0/0` en el puerto 22 significa que cualquier IP del planeta puede intentar conectarse
a tu servidor por SSH. Los bots de fuerza bruta escanean Internet constantemente buscando
exactamente esto. La buena práctica es restringir SSH siempre a **My IP** (tu IP actual) o,
mejor aún, a la IP de una VPN o un bastion host. En entornos de desarrollo puede ser tolerable
por conveniencia, pero en producción es un riesgo inaceptable.
:::

## Crear un Security Group

Aunque puedes configurar SGs durante el lanzamiento de una instancia, lo más común en la
práctica es crearlos de forma independiente desde su propio panel. Esto te permite diseñarlos
con calma, reutilizarlos en múltiples instancias y mantenerlos organizados. Lo encuentras en
el menú lateral de EC2, bajo **Network & Security → Security Groups**:

![Lista de Security Groups](/assets/02-compute/02-security/sg-list.png)

Aquí verás todos los SGs de la región, incluyendo el `default` que AWS crea automáticamente
en cada VPC. El botón **Create security group** (resaltado en rojo) abre el formulario de
creación:

![Formulario de creación de SG](/assets/02-compute/02-security/sg-create-form.png)

El formulario tiene tres secciones:

**Basic details** — El nombre del SG (`MyWebServerGroup` en el ejemplo), una descripción que
explique su propósito, y la VPC a la que pertenece. El nombre no puede modificarse después de
crearlo, así que elige uno descriptivo desde el principio. Una buena práctica es nombrar el
SG según su función y el componente que protege: `api-server-sg`, `db-mysql-sg`,
`bastion-ssh-sg`. Cuando tienes decenas de SGs en un proyecto real, un nombre genérico como
`sg-1` se convierte en un dolor de cabeza de mantenimiento.

**Inbound rules** — Las reglas de tráfico entrante. Por defecto está vacío: ningún tráfico
puede entrar. Tú agregas explícitamente lo que quieres permitir. Aquí aplica directamente
el **principio de mínimo privilegio**: habilita únicamente los puertos que tu aplicación
necesita y restringe las IPs de origen al rango más pequeño posible. Cada puerto abierto
innecesariamente es una superficie de ataque potencial.

**Outbound rules** — Las reglas de tráfico saliente. Por defecto viene con una regla que
permite **todo el tráfico de salida** hacia cualquier destino (`0.0.0.0/0`). AWS muestra un
aviso recomendando restringirlo, aunque en la mayoría de casos prácticos se deja así.

**Tags** — Etiquetas opcionales para identificar y filtrar el recurso en la consola y en la factura.

::: tip SGs en otros servicios de cómputo
Los Security Groups no son exclusivos de EC2. El mismo mecanismo se aplica en otros servicios
de cómputo y red: ECS (contenedores), RDS (bases de datos), Lambda (cuando se conecta a una
VPC), y Load Balancers. Si entiendes SGs en EC2, ya sabes cómo funcionan en el resto.
:::

## Anatomía de una Regla

Cada regla en un Security Group tiene la misma estructura, ya sea de entrada o de salida:

![Campos de una regla](/assets/02-compute/02-security/sg-rules-fields.png)

Una regla no es más que un **permiso**: dice exactamente qué tipo de tráfico, por qué protocolo,
en qué puerto y desde/hacia dónde se permite el paso. Podemos dividir una regla en dos partes:
**el puerto** y **el origen o destino**.

### La Parte del Puerto: Type, Protocol y Port Range

**Type** es el punto de entrada más amigable: un menú desplegable con los protocolos y puertos
más comunes ya predefinidos. Al seleccionar un tipo conocido, el protocolo y el puerto se
rellenan automáticamente:

![Tipo HTTPS con puerto predefinido](/assets/02-compute/02-security/sg-type-https.png)

Al elegir HTTPS, AWS completa automáticamente TCP como protocolo y 443 como puerto — porque
HTTPS siempre corre sobre TCP en el puerto 443. Esto evita errores de configuración en los
casos más frecuentes. Los tipos predefinidos más usados son SSH (TCP/22), HTTP (TCP/80),
HTTPS (TCP/443), MySQL/Aurora (TCP/3306), PostgreSQL (TCP/5432) y RDP (TCP/3389).

Cuando ningún tipo predefinido se ajusta a tu necesidad — por ejemplo, una aplicación que
escucha en un puerto personalizado — usas **Custom TCP** o **Custom UDP** y defines el puerto
manualmente:

![Tipo Custom con puerto editable](/assets/02-compute/02-security/sg-type-custom.png)

Con Custom TCP el campo *Port range* queda editable. Puedes especificar un puerto único
(`8080`), un rango (`8080-8090`) o múltiples puertos separados por coma.

### La Parte del Origen/Destino: Source y Destination

Con el puerto definido, el segundo elemento de la regla es de *dónde* viene el tráfico
(en reglas inbound) o *hacia dónde* va (en reglas outbound). El selector *Source* ofrece
cuatro opciones:

![Opciones de Source](/assets/02-compute/02-security/sg-source-options.png)

**Custom** — Defines manualmente una IP o rango de IPs en formato CIDR. Es la opción más
flexible y la más usada cuando necesitas precisión.

**Anywhere-IPv4** — Equivale a `0.0.0.0/0`: cualquier dirección IPv4 del mundo puede pasar.
Útil para servicios públicos (un servidor web), peligroso para servicios internos (SSH, bases
de datos).

**Anywhere-IPv6** — Lo mismo que el anterior pero para direcciones IPv6 (`::/0`).

**My IP** — AWS detecta automáticamente tu IP pública actual y la rellena. Conveniente para
dar acceso SSH solo desde donde estás trabajando ahora mismo, aunque cambia si tu IP es dinámica.

### CIDR: El Sistema de Rangos de IPs

Cuando eliges *Custom*, el campo de búsqueda muestra las opciones disponibles para especificar
el origen:

![Opciones CIDR y SGs como fuente](/assets/02-compute/02-security/sg-cidr-options.png)

El panel se divide en tres secciones. Los **CIDR blocks** son rangos de IPs expresados en
notación CIDR — un sistema que usa una barra y un número para indicar cuántas IPs incluye el
rango. `0.0.0.0/0` significa todas las IPs posibles. `192.168.1.0/24` significa las 256 IPs
del rango 192.168.1.0 a 192.168.1.255. Cuanto mayor el número tras la barra, más específico
y reducido es el rango — `/32` es exactamente una sola IP.

Los **Security Groups** en esta sección permiten usar otro SG como origen — lo que abre una
posibilidad muy poderosa que merece su propia explicación.

Los **Prefix lists** son listas mantenidas por AWS con rangos de IPs de sus propios servicios
(como CloudFront o S3), útiles para permitir tráfico solo desde esos servicios sin tener que
gestionar manualmente listas de IPs que cambian con el tiempo.

### Usar un Security Group como Origen

En lugar de especificar IPs, puedes indicar que el tráfico de entrada proviene de *cualquier
recurso que tenga asignado un SG específico*. Esta es una de las funcionalidades más elegantes
de los Security Groups porque desacopla la seguridad de las IPs — que pueden cambiar — y la
asocia a roles lógicos de tu arquitectura.

![SG como fuente de tráfico entre instancias](/assets/02-compute/02-security/sg-as-source.png)

El diagrama ilustra el patrón perfectamente: tienes dos instancias, una API (SG1) y una base
de datos MySQL (SG2). No quieres que la base de datos sea accesible desde Internet — solo desde
la API. En lugar de hardcodear la IP de la API (que cambia si la instancia se reinicia), configuras
SG2 para permitir entrada en el puerto 3306 únicamente desde SG1. Si mañana escala la API a
diez instancias, todas con SG1 asignado, todas podrán conectarse a la base de datos sin tocar
ninguna regla.

```
SG1 (API):                        SG2 (Base de datos):
  Inbound:                          Inbound:
    Puerto 80, 443 ← 0.0.0.0/0       Puerto 3306 ← SG1  ← solo la API
    Puerto 22      ← Mi IP
```

Este ejemplo también ilustra otra buena práctica: **separar responsabilidades en SGs distintos**.
Un SG por componente de tu arquitectura — uno para el servidor web, otro para la base de datos,
otro para el bastion de SSH — hace que cada pieza sea independiente, reutilizable y fácil de
auditar. Si en el futuro necesitas cambiar las reglas de acceso a la base de datos, modificas
solo SG2 sin tocar nada más. Mezclar todo en un único SG con docenas de reglas convierte la
seguridad en un laberinto imposible de mantener.

Además, usar otro SG como source es siempre preferible a usar una IP cuando el tráfico viene
de recursos dentro de AWS. Las IPs de las instancias cambian al reiniciarlas y los rangos CIDR
de una VPC pueden solaparse entre proyectos. Un SG como fuente es estable, legible e inequívoco.

## Crear un Security Group desde la CLI

La CLI permite crear y configurar Security Groups con la misma precisión que la consola, con
la ventaja de que los comandos son reproducibles y automatizables. Todo lo que hacemos aquí
puede convertirse en un script que reconstruye tu infraestructura de red en segundos.

::: tip La región siempre importa
Antes de crear cualquier recurso desde la CLI, verifica en qué región estás operando. A diferencia
de la consola web (que muestra la región en la barra superior), la CLI trabaja silenciosamente
con la región configurada por defecto y no te avisa si estás en el lugar equivocado. Consulta
la sección [Gestión de Regiones desde la CLI](/transversal/regiones-cli) para ver cómo verificar
y cambiar de región rápidamente.
:::

### Crear el Security Group

```bash
aws ec2 create-security-group \
  --group-name web-group \
  --description "SG for Web server" \
  --region eu-west-1
```

Cada parámetro tiene su propósito: `--group-name` es el nombre único del SG dentro de la VPC
(no puede repetirse en la misma VPC), `--description` es obligatorio y debe explicar el
propósito del grupo, y `--region` especifica explícitamente la región donde se creará. Si
no indicas la región, se usa la configurada por defecto en `~/.aws/config`.

![Resultado de create-security-group](/assets/02-compute/02-security/sg-cli-create-result.png)

La respuesta devuelve el `GroupId` (identificador único del SG, con formato `sg-xxxxxxxx`) y
el ARN completo del recurso. Guarda el `GroupId` — lo necesitarás para referenciar este SG
en otros comandos. Puedes también usar el nombre en algunos comandos con `--group-name`, pero
el ID es más seguro porque el nombre puede cambiar.

**Variaciones útiles:**

```bash
# Crear el SG en una VPC específica (no la default)
aws ec2 create-security-group \
  --group-name api-sg \
  --description "SG for API servers" \
  --vpc-id vpc-0abc12345 \
  --region eu-west-1

# Capturar el GroupId directamente en una variable
SG_ID=$(aws ec2 create-security-group \
  --group-name web-group \
  --description "SG for Web server" \
  --query 'GroupId' \
  --output text \
  --region eu-west-1)

echo "SG creado: $SG_ID"
```

Puedes verificar inmediatamente en la consola que el SG aparece en la lista:

![SG web-group en la consola](/assets/02-compute/02-security/sg-cli-console-list.png)

### Agregar Reglas de Entrada

Con el SG creado, el siguiente paso es definir qué tráfico puede entrar. El comando es
`authorize-security-group-ingress` y funciona igual para cualquier protocolo y puerto:

```bash
# Permitir tráfico HTTP (puerto 80) desde cualquier IP
aws ec2 authorize-security-group-ingress \
  --group-name web-group \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0 \
  --region eu-west-1
```

![Respuesta al agregar regla HTTP](/assets/02-compute/02-security/sg-cli-rule-http.png)

La respuesta confirma la regla creada: protocolo TCP, puerto 80, origen `0.0.0.0/0` (Internet
completo). El campo `IsEgress: false` confirma que es una regla de entrada (inbound).

**Variaciones — usar `--group-id` en lugar de `--group-name`:**

```bash
# Con el ID del SG (más robusto, funciona siempre)
aws ec2 authorize-security-group-ingress \
  --group-id sg-00abde1f542cae92c \
  --protocol tcp \
  --port 443 \
  --cidr 0.0.0.0/0 \
  --region eu-west-1

# Abrir un rango de puertos
aws ec2 authorize-security-group-ingress \
  --group-id sg-00abde1f542cae92c \
  --protocol tcp \
  --port 8080-8090 \
  --cidr 0.0.0.0/0 \
  --region eu-west-1
```

Usar `--group-name` es más legible pero solo funciona en la VPC por defecto. Si el SG está
en una VPC personalizada, debes usar `--group-id` obligatoriamente.

### Permitir SSH Solo desde Tu IP

Para SSH, la buena práctica es restringir el acceso a tu IP actual. El truco es obtener
tu IP pública automáticamente en el mismo comando usando un servicio externo:

```bash
# Linux / macOS
aws ec2 authorize-security-group-ingress \
  --group-name web-group \
  --protocol tcp \
  --port 22 \
  --cidr $(curl -s https://checkip.amazonaws.com)/32 \
  --region eu-west-1
```

```powershell
# Windows (PowerShell)
aws ec2 authorize-security-group-ingress `
  --group-name web-group `
  --protocol tcp `
  --port 22 `
  --cidr "$($(Invoke-WebRequest -Uri 'https://checkip.amazonaws.com' -UseBasicParsing).Content.Trim())/32" `
  --region eu-west-1
```

El `$(curl -s https://checkip.amazonaws.com)` consulta un endpoint de AWS que devuelve tu IP
pública en texto plano, y el `/32` al final convierte esa IP en un CIDR de un solo host — es
decir, exactamente tu IP y ninguna otra. En PowerShell, `Invoke-WebRequest` hace lo mismo pero
devuelve un objeto, por lo que hay que extraer el `.Content.Trim()` para obtener solo el texto.

![Respuesta al agregar regla SSH](/assets/02-compute/02-security/sg-cli-rule-ssh.png)

Verificando en la consola, el SG ahora tiene las dos reglas activas — SSH restringido a tu
IP y HTTP abierto a todo Internet:

![Inbound rules del SG web-group](/assets/02-compute/02-security/sg-cli-inbound-rules.png)


## El Comportamiento por Defecto y el Tráfico de Respuesta

Los Security Groups tienen dos comportamientos por defecto que hay que entender bien:

**Sin reglas de entrada, todo el tráfico entrante se bloquea.** El siguiente ejemplo muestra un
SG llamado "Disallow All" sin ninguna regla configurada — ningún tráfico puede entrar ni salir:

![SG sin reglas — bloqueo total](/assets/02-compute/02-security/sg-disallow-all.png)

Esto es importante: los SGs funcionan con una lógica de **permitir explícitamente** — si no
hay una regla que diga "esto pasa", no pasa. No existen reglas de bloqueo explícito porque
todo lo que no está permitido ya está bloqueado por defecto. No puedes agregar una regla que
diga "bloquear el puerto X" — simplemente no agregas una regla que lo permita.

**Los Security Groups son stateful.** Este es uno de los conceptos que más confunde al principio.
Imagina que configuras una regla de entrada que permite HTTP (puerto 80) pero no defines ninguna
regla de salida. Podrías pensar que el servidor puede recibir la petición HTTP pero no puede
enviar la respuesta — lo que haría el servicio completamente inútil. Pero no es así.

Los SGs rastrean el estado de las conexiones: cuando permiten entrar una petición, automáticamente
permiten salir la respuesta de esa misma conexión, sin necesidad de una regla de salida explícita.
Es la diferencia entre un firewall *stateful* (que recuerda el contexto de la conexión) y uno
*stateless* (que evalúa cada paquete de forma independiente). Los SGs son stateful; las Network
ACLs de VPC son stateless — pero eso es territorio de un módulo más avanzado de redes.

::: tip Audita tus Security Groups periódicamente
Con el tiempo los proyectos acumulan SGs huérfanos (sin instancias asociadas), reglas que
se abrieron "temporalmente" y nunca se cerraron, o permisos más amplios de lo necesario.
Una buena práctica es revisar periódicamente los SGs desde **EC2 → Network & Security →
Security Groups** y eliminar reglas o SGs que ya no tienen propósito. AWS también ofrece
**IAM Access Analyzer** y **AWS Trusted Advisor** para detectar configuraciones de red
excesivamente permisivas de forma automática.
:::

[docs.aws.amazon.com/vpc/latest/userguide/vpc-security-groups.html](https://docs.aws.amazon.com/vpc/latest/userguide/vpc-security-groups.html)
:::

## Key Pairs

Si los Security Groups controlan qué tráfico llega a tu servidor, los **Key Pairs** (pares de
claves) controlan quién puede entrar. Son el mecanismo de autenticación para conectarte a una
instancia EC2 por SSH — y a diferencia de una contraseña, son prácticamente imposibles de
adivinar por fuerza bruta.

La analogía más clara es la de una cerradura y una llave física: la instancia EC2 tiene la
cerradura instalada (la **clave pública**) y tú guardas la única llave que la abre (la
**clave privada**, el archivo `.pem` que descargaste). Cualquiera puede ver la cerradura,
pero sin la llave física no puede abrirla. AWS nunca guarda tu clave privada — en el momento
en que la generas, la descarga a tu computador y ya no existe en ningún servidor de AWS. Si
la pierdes, no hay forma de recuperarla: deberás generar un nuevo Key Pair.

Por debajo, los Key Pairs funcionan con **criptografía asimétrica**: se generan dos claves
matemáticamente relacionadas, donde lo que una cifra solo la otra puede descifrar. Cuando te
conectas por SSH, tu cliente presenta la clave privada y el servidor verifica que corresponde
a la clave pública que tiene instalada, todo sin que la clave privada viaje por la red.

### RSA vs ED25519 — ¿Cuál elegir?

AWS ofrece dos algoritmos de cifrado para los Key Pairs:

**RSA** — El estándar histórico, compatible con absolutamente todos los clientes SSH, incluyendo
versiones antiguas de OpenSSH, PuTTY y el cliente SSH nativo de Windows. Si tienes dudas,
usa RSA — funciona en cualquier entorno sin configuración adicional.

**ED25519** — Un algoritmo más moderno, con claves más cortas y operaciones matemáticas más
rápidas y seguras. Es la opción preferida en Linux y macOS con versiones recientes de OpenSSH.
Sin embargo, **en Windows puede no funcionar** con versiones antiguas del cliente SSH integrado
o con PuTTY sin configuración especial. Si trabajas principalmente desde Windows, quédate con RSA.

::: warning ED25519 y Windows
El cliente SSH incluido en versiones antiguas de Windows 10 y el PuTTY sin la librería
correspondiente no reconocen claves ED25519. El síntoma es un error de autenticación que
no dice claramente cuál es el problema. Si en algún momento SSH te rechaza la clave y no
entiendes por qué, verifica el tipo — si es ED25519, prueba regenerando con RSA.
:::

### Crear un Key Pair desde la Consola

Ya vimos cómo crear un Key Pair durante el lanzamiento de una instancia. También puedes
crearlos de forma independiente desde **EC2 → Network & Security → Key Pairs**:

![Lista de Key Pairs](/assets/02-compute/02-security/keypair-list.png)

El botón **Create key pair** (resaltado en naranja) abre el mismo formulario que ya conocemos:

![Formulario de creación de Key Pair](/assets/02-compute/02-security/keypair-create-form.png)

El formulario pide el nombre, el tipo (RSA o ED25519) y el formato del archivo de clave privada:
`.pem` para OpenSSH (el standard en Linux, macOS y Windows moderno) y `.ppk` para PuTTY.
Al hacer clic en *Create key pair*, el archivo se descarga automáticamente — es la única
oportunidad de descargarlo.

### Crear un Key Pair desde la CLI

```bash
aws ec2 create-key-pair \
  --key-name key-test \
  --key-type rsa \
  --query 'KeyMaterial' \
  --output text > key-test.pem
```

![Comando de creación de Key Pair vía CLI](/assets/02-compute/02-security/keypair-cli-create.png)

Este comando merece una explicación detallada porque usa varias opciones de la CLI en conjunto:

`--key-name key-test` — El nombre identificador del Key Pair en AWS. Con este nombre lo
referenciarás al lanzar instancias.

`--key-type rsa` — El algoritmo criptográfico. Puede ser `rsa` o `ed25519`.

`--query 'KeyMaterial'` — Filtra la respuesta JSON para extraer únicamente el campo
`KeyMaterial`, que contiene el contenido de la clave privada. Sin este filtro, la respuesta
incluye también el nombre, el ID y el fingerprint, y todo eso mezclado en el archivo `.pem`
lo corrompería.

`--output text` — Fuerza la salida como texto plano en lugar de JSON. Crucial aquí: si usas
`--output json`, el contenido de la clave tendría comillas y escapes que lo harían inválido.

`> key-test.pem` — El operador `>` redirige la salida estándar del comando hacia el archivo
`key-test.pem` en lugar de mostrarla en pantalla. Así la clave privada queda guardada
directamente en el archivo sin pasar por el portapapeles ni por la pantalla.

### Lanzar una Instancia con SG y Key Pair desde la CLI

Con el Security Group y el Key Pair listos, podemos lanzar una instancia que use ambos:

```bash
aws ec2 run-instances \
  --image-id ami-09c201095c86b2f893 \
  --instance-type t3.micro \
  --key-name key-test \
  --security-groups web-group \
  --count 1 \
  --region eu-west-1
```

![Respuesta de run-instances](/assets/02-compute/02-security/keypair-cli-run-instance.png)

La respuesta JSON confirma que la instancia se está creando con el SG `web-group` asignado.
Los campos más relevantes son `InstanceId` (el ID único de la instancia), `State` (iniciará

**Variaciones — por ID en lugar de nombre, y múltiples SGs:**

```bash
# Usar IDs en lugar de nombres (necesario en VPCs no-default)
aws ec2 run-instances \
  --image-id ami-09c201095c86b2f893 \
  --instance-type t3.micro \
  --key-name key-test \
  --security-group-ids sg-00abde1f542cae92c \
  --subnet-id subnet-0abc12345 \
  --region eu-west-1

# Asignar múltiples Security Groups simultáneamente
aws ec2 run-instances \
  --image-id ami-09c201095c86b2f893 \
  --instance-type t3.micro \
  --key-name key-test \
  --security-group-ids sg-00abde1f542cae92c sg-0f9e76b83ceff647d \

```

Para asignar múltiples SGs, simplemente listas los IDs separados por espacio después de
`--security-group-ids`. Con `--security-groups` (nombres) también funciona igual.

### Permisos del Archivo `.pem` — Un Error Frecuente

SSH rechaza claves privadas que cualquier usuario del sistema puede leer — es una medida de
seguridad del protocolo. Si el archivo `.pem` tiene permisos demasiado abiertos, SSH se niega
a usarlo con un error del tipo *"WARNING: UNPROTECTED PRIVATE KEY FILE!"*.

**En Linux y macOS**, la solución es directa:

```bash
chmod 400 key-test.pem    # solo el propietario puede leer, nadie más
ssh -i key-test.pem ec2-user@<ip>
```

**En Windows con PowerShell**, los permisos del sistema de archivos NTFS funcionan diferente.
La forma más directa es usar el comando `icacls`:

```powershell
# Eliminar todos los permisos heredados y dejar solo el tuyo
icacls key-test.pem /inheritance:r /grant:r "$($env:USERNAME):(R)"
```

O desde la interfaz: clic derecho en el archivo → Propiedades → Seguridad → Editar → elimina
todos los usuarios excepto el tuyo y deja solo lectura.

### El Problema de LF vs CRLF — No Abras el `.pem` con Notepad

::: danger No edites el archivo .pem con editores de texto de Windows
Si abres el archivo `key-test.pem` con **Notepad**, **Word** u otros editores de Windows,
pueden ocurrir dos problemas que corrompen la clave silenciosamente:

**1. Cambio de saltos de línea (LF → CRLF):** Los archivos `.pem` usan saltos de línea Unix
(`LF`, `\n`). Windows Notepad puede convertirlos a estilo Windows (`CRLF`, `\r\n`), lo que
hace que la clave sea inválida — SSH no puede parsearla correctamente.

**2. Codificación incorrecta:** El archivo debe estar en **UTF-8 sin BOM** (Byte Order Mark).
Algunos editores agregan un BOM invisible al principio del archivo que corrompe el header
de la clave PEM.

Si necesitas inspeccionar el contenido del archivo, usa **VS Code**, **Notepad++** (con
codificación UTF-8 sin BOM y saltos LF) o simplemente `cat key-test.pem` en la terminal.
:::
